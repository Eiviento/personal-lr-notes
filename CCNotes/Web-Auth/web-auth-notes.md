## 1. 问题起源：HTTP 是无状态的

```
你第一次请求：  GET /page   → 服务器："你是谁？"
你第二次请求：  GET /page   → 服务器："你是谁？"
```

**HTTP 协议本身不记得上次是谁发过请求。** 但 Web 应用需要知道"这个人登录过吗？他的购物车里有什么？"

核心矛盾：**浏览器和服务器之间需要一个"身份证"机制**。

---

## 2. 第一代：Cookie 直接存数据

### 思路

> "把数据放在浏览器里，每次请求带过来就行了。"

```
服务器 → 浏览器：  Set-Cookie: username=张三; points=120
浏览器 → 服务器：  Cookie: username=张三; points=120
服务器："好的张三，120 分，我信你。"
```

### 优点

- 服务器不需要存任何东西，无状态，省资源。

### 致命缺陷

```
你把 Cookie 改成：  username=张三; points=99999
服务器："好的张三，99999 分 ✅"

—— 服务器无法校验，照单全收。
```

### 结论

**纯 Cookie 存数据 = 让用户自己保管凭证且不防伪，完全不安全。**

---

## 3. 第二代：Session 服务端存数据

### 思路

> "数据我来存，只给你一个随机编号当钥匙。"

```
登录：

  浏览器 ── "我是张三，密码xxx" ──→ 服务器
                                        │
                                        │ 创建 Session 记录：
                                        │ "A38F2C" → {username:"张三", id:7, points:120}
                                        │
  浏览器 ←── Set-Cookie: session_id=A38F2C ──

后续请求：

  浏览器 ── Cookie: session_id=A38F2C ──→ 服务器
                                              │
                                              │ 查内存/数据库：
                                              │ "A38F2C" → 张三
                                              │
  浏览器 ←── 张三的数据 ──────────────────
```

### 优点

- 数据在服务端，用户改不了
- 随时可以服务端删除 Session 踢人下线

### 缺点

- 服务器要存所有在线用户的状态，用户多了压力大
- 多台服务器需要共享 Session（Redis 等），扩展麻烦

### 对比

| | 纯 Cookie | Session |
|---|---|---|
| **数据存在哪** | 浏览器 | 服务器 |
| **浏览器拿什么** | 真实数据 | 一个随机 ID |
| **用户能改吗** | 能 | 不能（改了 ID 查不到） |
| **服务器压力** | 无 | 有（存 Session） |

## 4. 第三代：JWT 带签名自包含

### 思路

> "数据还是让用户带着，但我给数据盖个防伪章，一验就知道有没有被改过。"

### 比喻

| 方式 | 比喻 |
|------|------|
| 纯 Cookie | 手写纸条，随便改 |
| Session | 店里存档，只给顾客一个手环编号 |
| JWT | 塑封防伪会员卡，信息写在卡上但没法涂改 |

### 工作原理

```
服务器生成 JWT：

  数据：  {"name":"张三", "points":120, "exp":"2026-08-12"}
          ↓
  用密钥对数据算签名： HMAC(数据 + 密钥) → "dGhpcyBp..."
          ↓
  JWT = Base64(数据) + "." + Base64(签名)

用户下次请求带着 JWT：

  服务器收到 JWT
    → 用同样的密钥重新算一遍签名
    → 算出来一样 → 数据没被改过 ✅
    → 算出来不一样 → 被篡改了 ❌
```

### 核心优势：无状态 + 分布式友好

```
Session 方案（多服务器）：
  服务器A ←→ Redis/DB ←→ 服务器B
  需要共享 Session 数据

JWT 方案（多服务器）：
  服务器A：自己验证签名，不用查
  服务器B：自己验证签名，不用查
  各干各的，不需要通信
```

### 劣势

- **发出去了就难收回**：Session 可以服务端删掉立即踢人；JWT 在有效期内一直在用户手里
- **别存敏感信息**：JWT 的 Body 只是 Base64 编码（不是加密），任何人都能解码看内容
- **体积大**：每次请求都带着，比 Session ID 大多了

### 三种方案对比

| | 纯 Cookie | Session | JWT |
|---|---|---|---|
| **数据在哪** | 浏览器 | 服务器 | 浏览器（带签名） |
| **能篡改吗** | 能 | 不能 | 不能（签名验证） |
| **服务器存东西吗** | 不存 | 存 | 不存 |
| **能随时撤回吗** | — | 能 | 难（需要黑名单） |
| **多服务器友好** | 好 | 差 | 极好 |

### 工业级实践：双令牌机制

```
Access Token（短命）
  ├─ 有效期：15 分钟
  ├─ 每次请求都带着
  └─ 被偷了 → 15 分钟后自动作废

Refresh Token（长寿）
  ├─ 有效期：7 天
  ├─ 只在"换新 Access Token"时用一下
  ├─ 存在 HttpOnly Cookie 里
  └─ 可以被服务端主动撤销

流程：
  登录 → 拿到 Access Token + Refresh Token
  Access Token 过期 → 用 Refresh Token 换新的
  Refresh Token 被拉黑 → 最多 15 分钟就彻底失效
```

## 5. Cookie 的安全属性

Cookie 本身只是传输载体。它的安全性取决于属性设置。

### 核心属性

```
Set-Cookie: session_id=A38F2C; HttpOnly; Secure; SameSite=Lax
                ↑                  ↑         ↑           ↑
              数据本身          JS不可读   只走HTTPS   防跨站请求
```

### 各属性防什么

| 属性 | 防什么 | 原理 |
|------|--------|------|
| **HttpOnly** | XSS 脚本偷 Cookie | `document.cookie` 读不到 |
| **Secure** | Cookie 在 HTTP 明文传输 | 浏览器只在 HTTPS 连接时发送 |
| **SameSite** | 跨站请求伪造（CSRF） | 从别的网站发起的请求不带上 Cookie |

### HttpOnly 为什么 JS 读不到？

不是加密，是**浏览器强制执行**的规则：

```
普通 Cookie：
  document.cookie → "username=张三; session_id=A38F2C"
  JS 全能看到

HttpOnly Cookie：
  document.cookie → "username=张三"
  session_id 被浏览器藏起来了，JS 根本不知道它存在
  （但每次发请求，浏览器会自动带上它）
```

> ⚠️ HttpOnly 防的是"恶意脚本在你的浏览器里跑"（XSS），不防网络抓包。

## 6. 传输层安全：TLS 加密管道

### 为什么必须要有 TLS？

**没有 HTTPS，上面所有机制都在裸奔：**

```
HTTP（明文）：

  你 ──── GET / ──────────────────→ 服务器
         Cookie: session_id=A38F2C
         ↑
    星巴克 Wi-Fi 上任何一个人打开 Wireshark：

    GET / HTTP/1.1
    Host: example.com
    Cookie: session_id=A38F2C    ← 明文可见，复制就能冒充你！
```

### TLS 做了什么

```
浏览器                                   服务器
  │                                         │
  │ ① "我会这些加密算法"  ───────────────→  │
  │                                         │
  │ ← ② 证书 + ECDH 公钥 ────────────────   │
  │                                         │
  │ ③ 验证证书（CA 签名链）                  │
  │ ④ ECDH 密钥交换                         │
  │                                         │
  │ ═══ 协商出会话密钥，开始加密 ═══════════ │
  │                                         │
  │ 此后所有数据都用会话密钥加密传输          │
```

### TLS 防住了什么

| 攻击 | 怎么防的 |
|------|---------|
| **偷听** | 数据全加密，没有会话密钥解不开 |
| **篡改** | 每段数据带 MAC（消息认证码），改一个比特都会被发现 |
| **冒充** | 证书 + CA 签名链，假网站拿不到真证书 |
| **重放** | 每次连接的会话密钥都是新的，上次的密文这次用不了 |

## 7. TLS 的核心：ECDH 密钥交换

### 问题

客户端和服务器在**公开信道**（可能被监听）上，怎么商量出一把**只有他俩知道的密钥**？

### ECDH = 椭圆曲线版 DH 密钥交换

以颜色做比喻：

```
公开的黄色 —— 所有人都能看到

客户端：                        服务器：
  私密颜色（红）                  私密颜色（蓝）
  + 黄色                         + 黄色
  = 橙色（公开传给服务器）          = 绿色（公开传给客户端）

收到对方的绿色                    收到对方的橙色
  + 自己的红色                     + 自己的蓝色
  = 棕色 ✅                       = 棕色 ✅

结果：双方都得到了棕色
但网络上只传了橙色和绿色
从橙色+绿色反推不出棕色
```

### 数学上

```
双方约定：一条椭圆曲线 + 基点 G

客户端生成随机数 a            服务器生成随机数 b
A = G × a                    B = G × b

交换 A 和 B

客户端算：B × a               服务器算：A × b
       = (G × b) × a               = (G × a) × b
       = G × (a × b) ✅           = G × (a × b) ✅

安全性基于椭圆曲线离散对数难题：
已知 G 和 G×a，反推 a → 极其困难
```

### ECDH vs 传统 DH

| | DH | ECDH |
|---|---|---|
| **底层数学** | 模幂运算 | 椭圆曲线点乘 |
| **同等安全的密钥长度** | 2048 位 | 256 位 |
| **速度** | 慢 | 快很多 |
| **适用** | 传统服务器 | 移动端、IoT、TLS 1.3 |

> TLS 1.3 默认用 ECDH，最常用曲线是 **X25519**。

## 8. 密钥派生：从 Shared Secret 到干活的多把钥匙

ECDH 出来的只是一个"原始材料"，还需要派生出一整套密钥。

### 密码学原则：一把钥匙只干一件事

```
实际通信需要：

  1. 加密数据       → 加密密钥
  2. 防篡改         → MAC 密钥
  3. 客户端→服务端   → 客户端 IV（初始向量）
  4. 服务端→客户端   → 服务端 IV
```

### 完整派生流水线

```
ECDH 完成
    │
    ▼
 Shared Secret（共享秘密）
   ECDH 的原始产物：G×(a×b)
   不一定熵均匀，不够完美
    │
    │  + Client Random + Server Random
    │  （握手时双方交换的随机数）
    ▼
 Master Secret（主密钥）
   把 Shared Secret + 随机数 一起搅拌（PRF/HKDF）
   产出熵均匀的主密钥
   即使 ECDH 产物有偏差，这一步也能救回来
    │
    │  HKDF-Expand
    ▼
┌───────┬───────┬───────┬───────┐
│ 加密   │ MAC   │ 客户端 │ 服务端 │
│ 密钥   │ 密钥  │  IV   │  IV   │
└───────┴───────┴───────┴───────┘
  加密     防篡改    客户端    服务端
                     →服务端    →客户端
```

### 两层结构的意义

| 层 | 角色 | 比喻 |
|---|---|---|
| **Shared Secret** | ECDH 的原始产物 | 从地里挖出来的原油 |
| **Master Secret** | 加入随机数，搅拌提纯 | 炼油厂加工后的汽油 |
| **加密/MAC/IV 密钥** | 最终干活用的 | 注入发动机 |

> TLS 1.3 把这个过程标准化为 **HKDF**（HMAC-based Key Derivation Function）。

## 9. 全景串联：一次完整请求中的洋葱模型

### 从打开浏览器到服务器认出你

```
┌─────────────────────────────────────────────────────────────────┐
│  你在浏览器输入 https://example.com，登录，然后访问页面           │
└─────────────────────────────────────────────────────────────────┘

第一层：建立加密管道（TLS + ECDH）
─────────────────────────────────

  浏览器 ←→ 服务器
  
  ① ClientHello（支持的加密算法）
  ② 服务器发来证书 + ECDH 公钥 B
  ③ 浏览器验证证书（CA 签名链）
  ④ 浏览器生成 ECDH 密钥对，发公钥 A
  ⑤ 双方各自算出 Shared Secret = G×(a×b)
  ⑥ 派生 → Master Secret → 加密密钥 + MAC密钥 + 客户端IV + 服务端IV
  
  ═══════ 加密隧道建立 ═══════
  之后所有数据都被加密，外人看到的全是密文


第二层：证明身份（登录 → Session）
─────────────────────────────────

  加密管道内（所有内容被 TLS 加密）：
  
  浏览器 ── POST /login {"username":"张三","password":"****"} ──→ 服务器
                                                                      │
                                                                    验证密码
                                                                    创建 Session
                                                                    内存存："A38F2C" → {username:"张三",id:7}
  浏览器 ←── Set-Cookie: session_id=A38F2C; HttpOnly; Secure ─────  


第三层：后续每次请求（三层防护叠加）
─────────────────────────────────────

  加密管道内：

  浏览器自动行为：
    取出 Cookie → session_id=A38F2C
    塞进请求头

  浏览器 ── GET /my-orders ──────────────→ 服务器
          Cookie: session_id=A38F2C              │
          （加密管道内传输，外人看是密文）          │
                                                │ 拿到 session_id
                                                │ 查内存 → A38F2C = 张三
                                                │ 查出张三的订单
  浏览器 ←── 订单页面 HTML ──────────────────
  
  用户完全无感，浏览正常
```

### 攻击者视角（洋葱模型从外到内）

```
┌──────────────────────────────────────────────────────┐
│                                                      │
│  攻击者想拿到凭证冒充用户                              │
│                                                      │
│  ┌──────────────────────────────────────────────┐    │
│  │ TLS 层                                       │    │
│  │                                              │    │
│  │ Wi-Fi 抓包 → 全是密文 ❌ 拿不到              │    │
│  │ 中间人篡改 → MAC 验证不通过 ❌               │    │
│  │ 重放上次请求 → 会话密钥已换 ❌                │    │
│  │                                              │    │
│  └──────────────────────────────────────────────┘    │
│                         │                            │
│                    TLS 被绕过？                        │
│                    （恶意软件在你电脑上、物理看屏幕等）  │
│                         │                            │
│                         ▼                            │
│  ┌──────────────────────────────────────────────┐    │
│  │ Cookie 属性层                                 │    │
│  │                                              │    │
│  │ XSS 注入脚本 → HttpOnly 挡掉 ❌              │    │
│  │ HTTP 明文传输 → Secure 阻止发送 ❌            │    │
│  │ 跨站伪造请求 → SameSite 不发送 Cookie ❌     │    │
│  │                                              │    │
│  └──────────────────────────────────────────────┘    │
│                         │                            │
│                    拿到了 Cookie 值？                  │
│                     （设备被物理攻破）                  │
│                         │                            │
│                         ▼                            │
│  ┌──────────────────────────────────────────────┐    │
│  │ 服务端层                                      │    │
│  │                                              │    │
│  │ 异常 IP / 设备指纹 → 拒绝 ❌                  │    │
│  │ 关键操作二次验证 → 还需要密码 ❌               │    │
│  │ Session 被服务端删除 → ID 作废 ❌             │    │
│  │                                              │    │
│  └──────────────────────────────────────────────┘    │
│                                                      │
│  每一层解决不同问题，没有哪层能单独保证安全             │
│  但合在一起让攻击成本变得极高                          │
│                                                      │
└──────────────────────────────────────────────────────┘
```

### 完整链路一句话

```
ECDH/TLS 建了一条加密管道
    ↓
管道里传 Cookie
    ↓
Cookie 里装 Session ID（或 JWT）
    ↓
HttpOnly + Secure + SameSite 保护 Cookie
    ↓
服务端拿到 Session ID → 查内存 → 认出你
```

## 10. 总结：各层职责一览

### 技术演进路线

```
纯 Cookie 存数据
  → 不安全，用户能篡改
       ↓
Session 服务端存
  → 安全了，但服务端压力大、多服务器扩展难
       ↓
JWT 带签名自包含
  → 安全 + 无状态 + 易扩展
  → 但发出去了难收回
       ↓
再加上 TLS
  → 补齐了网络安全这最关键的一环
```

### 各层防什么的最终结论

| 攻击方式 | 防住它的技术 |
|----------|-------------|
| 网络抓包偷凭证 | **TLS** |
| 网络抓包看内容 | **TLS** |
| 中间人篡改 | **TLS（MAC 验证）** |
| 重放攻击 | **TLS（每次连接新密钥）** |
| XSS 脚本偷 Cookie | **HttpOnly** |
| HTTP 明文传 Cookie | **Secure** |
| 跨站请求伪造 | **SameSite** |
| 用户篡改自己的身份数据 | **Session / JWT 签名** |
| 猜别人的 Session ID | **随机数够长** |
| 已知被盗，立刻踢人 | **Session 删除** |

### 一句话总结

```
Session / Cookie / JWT 解决的是"应用层身份管理"，
TLS 解决的是"网络层安全传输"。

没有 TLS，
上面的身份机制就是在广场上互相喊话——
喊的内容别人全能听见，听见了就能冒充你，Session 和 Cookie 对此完全无能为力。
```
