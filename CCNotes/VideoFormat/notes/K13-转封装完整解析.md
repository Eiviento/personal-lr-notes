# K13 — 转封装（Transmuxing）完整解析

> 学习日期：2026-07-11
> 前置依赖：K05

---

## 1. 定义

**转封装 = 换容器，不碰编码数据。** 容器变了，但里面的视频/音频编码流一个比特都不改。

---

## 2. 拆解：Demux → Mux

```
步骤 1 — Demux（拆箱）：
    MKV 容器 → 拆出：video.h264（裸流） + audio.aac（裸流）

步骤 2 — Mux（装箱）：
    裸流 → 装进 MP4 容器 → output.mp4（编码数据原封不动）
```

---

## 3. 为什么无损

编码层和封装层是独立维度（K05），转封装只操作封装层。

---

## 4. 转封装 vs 转码

| | 转封装 | 转码 |
|--|--------|------|
| 动了什么 | 只动容器 | 解码+重编码 |
| 编码格式 | 不变 | 可变 |
| 画质 | 无损（bit-exact） | 有损（再次量化） |
| 分辨率/帧率/码率 | 不能改 | 都能改 |
| 速度 | 极快（磁盘 I/O 极限） | 慢（CPU/GPU 编码） |
| FFmpeg 标志 | `-c copy` | `-c:v libx264` 等 |

---

## 5. FFmpeg 实战

```bash
# 全流转封装
ffmpeg -i input.mkv -c copy output.mp4

# 只保留视频
ffmpeg -i input.mkv -c:v copy -an output.mp4

# 选特定流
ffmpeg -i input.mkv -map 0:v:0 -map 0:a:1 -c copy output.mp4

# 视频不动，音频转码
ffmpeg -i input.mkv -c:v copy -c:a aac -b:a 128k output.mp4
```

---

## 6. 为什么不能总是转封装

新容器必须支持原编码格式。MKV→MP4 时，VP9/Opus/FLAC 等可能不兼容（K14 详讲）。

---

## 7. 典型场景

- 兼容播放器：MKV → MP4，3 秒完成，无损
- 直播录制转点播：FLV → MP4
- 提取裸流：MP4 → Annex B 裸 H.264

---

## 补充：为什么没有统一容器？

- MP4 为文件存储设计（有索引）
- TS 为不可靠广播传输设计（188 字节小包，丢了能恢复）
- MKV 为"容纳一切"设计（开源，什么都装得下）
- 各容器出生时有不同使命，加上专利/商业利益、未来需求不可预知
- FFmpeg 的意义：让这堆不统一的容器之间能自由转换
