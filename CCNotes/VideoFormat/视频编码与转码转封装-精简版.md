# 视频编码与转码/转封装 — 核心概念

---

## 一、为什么需要压缩

未压缩视频的数据量：

```
1080p @ 30fps, RGB 24-bit

一帧 = 1920 × 1080 × 3 字节 = 6.22 MB
一秒 = 30 × 6.22 MB = 186.6 MB  
码率 ≈ 1.5 Gbps

作为对比：在线 1080p 视频只有 5-10 Mbps
          差 150-300 倍 —— 这就是压缩的意义
```

压缩分两步走：**先压单帧内部 → 再压帧与帧之间**。

---

## 二、第一步：单帧内的压缩（RGB → YUV 4:2:0）

### 2.1 RGB → YUV：分离亮度与色度

人眼的生理特性：**对亮度（明暗）极度敏感，对色度（颜色细节）相对迟钝。**

利用这一点，把 RGB 转换成 YUV：

```
RGB：                         YUV：
├─ R 红色  ─┐                ├─ Y  亮度（Luma）    ← 黑白图像的全部信息，必须保留
├─ G 绿色  ─┼─ 三个通道同等重要  ├─ Cb 蓝色色度       ← "上色"数据，可以压缩
└─ B 蓝色  ─┘                └─ Cr 红色色度       ← "上色"数据，可以压缩
```

只看 Y 通道 = 一张完美的黑白照片。Cb/Cr 只是给这张黑白图"涂上颜色"。

### 2.2 色度采样：砍掉冗余的色度信息

既然人眼对色度不敏感，就让相邻像素**共享色度**：

```
4:4:4 — 每个像素都有独立 Y + Cb + Cr（无压缩）
4:2:2 — 水平方向每 2 个像素共享 1 组 Cb/Cr（减少 33%）
4:2:0 — 2×2 的 4 个像素共享 1 组 Cb/Cr（减少 50%，最常用）

你看到的所有在线视频、蓝光、手机录像 → 全是 4:2:0
```

**YUY2** 就是 4:2:2 的一种内存排列格式，常用于 USB 摄像头和采集卡的原始输出。它的定位是"采集中间格式"而非最终的存储格式。

### 小结

```
RGB(每像素24bit) → YUV 4:2:0(每像素12bit) = 数据量直接减半
这是第一道压缩，利用了人眼的生理特性，几乎不影响观看画质
```

## 三、第二步：连续帧的压缩（MJpeg vs H.264）

单帧内的压缩（色度采样）做完后，数据量依然巨大。接下来有两种思路处理连续帧：

### 3.1 MJpeg 的思路：每帧独立压缩

```
MJpeg = 把每一帧压缩成一张 JPEG 图片，然后串起来播放

帧1: [完整 JPEG] 帧2: [完整 JPEG] 帧3: [完整 JPEG] ...
      100 KB         100 KB         100 KB

每帧独立，帧与帧之间互不依赖。
```

JPEG 的压缩原理（逐帧执行）：
```
RGB → YUV → 色度采样 → 切成 8×8 块 → DCT 变换(空间域→频域) 
→ 量化(丢掉高频细节) → 熵编码(数学压缩)
```

**MJpeg 的本质特征**：
- 帧内编码（Intra-frame）：只消除单帧内的冗余
- 每帧独立 → 随机访问任意帧、丢一帧不影响其他帧、编辑精确到帧
- 代价：同样画质下文件是 H.264 的 **3-5 倍**（因为完全不利用帧间相似性）

**典型场景**：监控实时预览、无人机图传（需要极低延迟 + 信号中断后秒恢复）

### 3.2 H.264 的思路：利用帧间相似性

H.264 在 MJpeg 的基础上多了一步——**帧间预测**：

```
视频中相邻帧通常极其相似（背景不变，只有物体微动）。

MJpeg 的做法：
    帧1: [完整编码蓝天+完整编码白云在左边]  100 KB
    帧2: [完整编码蓝天+完整编码白云在右边]  100 KB  
    帧3: [完整编码蓝天+完整编码白云在右边]  100 KB
    总计：300 KB

H.264 的做法：
    帧1: [完整编码]  ← I 帧（Intra，独立帧）      100 KB
    帧2: [云从(x1→x2)，其他照旧]  ← P 帧          10 KB
    帧3: [和帧2一样，没变化]       ← P 帧          2 KB
    总计：112 KB
```

H.264 把帧分为三种类型：

| 帧类型 | 怎么编码 | 体积 | 作用 |
|--------|---------|------|------|
| **I 帧**（Intra） | 完整编码，类似 JPEG | 大 | 随机访问入口、错误恢复锚点 |
| **P 帧**（Predicted） | 只存与前一帧的差异 | ~I 的 1/5~1/10 | 日常压缩主力 |
| **B 帧**（Bidirectional） | 参考前后两帧的差异 | ~I 的 1/10~1/20 | 进一步压缩 |

一组 I+P+B 帧构成一个 **GOP**（Group of Pictures），GOP 从一个 I 帧开始到下一个 I 帧前结束。GOP 越长压缩越狠，但拖动进度条时越慢（需要回溯到最近 I 帧解码）。

H.264 编码流程：`预测（帧内/帧间） → 残差 → DCT变换 → 量化（有损） → 熵编码（无损）`

### 小结

```
MJpeg：只做帧内压缩，每帧独立，文件大但延迟低、编辑友好
H.264：帧内 + 帧间压缩，利用帧间相似性，文件小 3-5 倍，场景全覆盖
```

## 四、编码层 vs 封装层

编码完了，数据需要存成文件。这里涉及两个独立的概念：

```
编码层（Codec）：视频数据怎么压缩
    → H.264 / H.265 / MJpeg / VP9 / AV1

封装层（Container）：压缩后的数据怎么打包成文件  
    → MP4 / MKV / MOV / TS / AVI

一个 .mp4 文件 ≈ MP4 容器 + 里面装着 H.264 编码的视频流 + AAC 编码的音频流
常见误解：.mp4 = H.264 ← 错误，.mp4 是容器，里面的编码可以是 H.264、H.265 等
```

## 五、转封装 vs 转码

### 5.1 转封装（Transmuxing）

**只换容器，不动编码数据。** 拆开旧箱子，把东西原封不动装进新箱子。

```
MKV(H.264+AAC) → demux → 裸H.264流 + 裸AAC流 → mux → MP4(H.264+AAC)
                                                       ↑
                                            编码数据一个 bit 都没变
```

| 特点 | 说明 |
|------|------|
| 画质 | ✅ 无损（bit-exact） |
| 速度 | 秒级（磁盘 I/O 极限） |
| 能改什么 | 只能换容器格式 |
| 不能改什么 | 编码格式、分辨率、帧率、码率 |
| FFmpeg 命令 | `ffmpeg -i in.mkv -c copy out.mp4` |

### 5.2 转码（Transcoding）

**解码 → 重新编码。** 把压缩数据展开为原始像素，再按新参数压缩。

```
MP4(H.264) → 解码器 → YUV原始像素 → [可选：缩放/水印/裁剪] → 编码器 → MP4(H.265)
                ↑                                                      ↑
          展开为 ~3MB/帧的裸数据                              按新参数重新压缩
```

| 特点 | 说明 |
|------|------|
| 画质 | ❌ 有损（量化步骤不可逆，每转一次丢一次信息） |
| 速度 | 慢（CPU/GPU 编码，10分钟视频可能需3-15分钟） |
| 能改什么 | 编码格式、分辨率、帧率、码率、添加水印字幕... |
| 代价 | 代际损失——每转码一次画质就降一次 |
| FFmpeg 命令 | `ffmpeg -i in.mp4 -c:v libx264 -crf 23 out.mp4` |

### 5.3 怎么选

```
只换文件格式（MKV→MP4）？                          → 转封装，秒级无损
要改分辨率/编码格式/码率/加水印？                     → 必须转码
要快？                                              → 转封装
要省空间（H.264→H.265）？                           → 必须转码
```

## 六、三种格式总览

| | YUY2 | MJpeg | H.264 |
|--|------|-------|-------|
| 是什么 | 原始像素格式（4:2:2排列） | 帧内编码（逐帧JPEG） | 帧内+帧间编码 |
| 定位 | 采集链路的"中转站" | 低延迟预览/简单录制 | 一切最终存储和分发 |
| 压缩率 | 无（~1 Gbps @1080p） | 低（50-100 Mbps） | 高（5-10 Mbps） |
| 帧独立性 | 每帧独立 | 每帧独立 | P/B帧依赖I帧 |
| 延迟 | 零 | 毫秒级 | 取决于GOP长度 |
| 编辑友好 | — | 精确到帧 | 仅I帧处精准切割 |

## 七、压缩链路全景

```
原始光信号
    │
    ▼
┌─────────────────────────────────────────────┐
│ ① 像素采集：传感器 → YUY2 原始帧              │
│    数据量：1080p ≈ 1 Gbps                   │
│    这一步还没开始压缩                          │
├─────────────────────────────────────────────┤
│ ② 单帧压缩：RGB → YUV 4:2:0                 │
│    数据量：砍半（~746 Mbps）                  │
│    手段：利用人眼对色度不敏感                   │
├─────────────────────────────────────────────┤
│ ③ 连续帧压缩：帧内+帧间编码（H.264核心）        │
│    数据量：再砍 50-200 倍（~5-10 Mbps）        │
│    手段：帧内预测 + 帧间预测 + DCT量化 + 熵编码  │
├─────────────────────────────────────────────┤
│ ④ 封装：编码流 → MP4/MKV/TS 文件              │
│    容器和编码是两层，互相独立                    │
├─────────────────────────────────────────────┤
│ ⑤ 交付后处理：                                │
│    转封装（换容器，无损，秒级）                   │
│    转码（重编码，有损，慢，但什么都能改）          │
└─────────────────────────────────────────────┘
```
