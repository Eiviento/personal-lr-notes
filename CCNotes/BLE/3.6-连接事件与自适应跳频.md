# 3.6 连接事件与自适应跳频

## 一、Connection Event：连接后的心跳

```
        Conn Interval (如 30ms)
  ├─────────────────────────────────────┤
  │         Connection Event            │    休眠...
  │  Master → Slave  (数据包)           │
  │         ← IFS 150μs →              │
  │  Slave → Master  (回复包)           │
  │         ← IFS 150μs →              │
  │  Master → Slave  (可再来多轮)        │
  └─────────────────────────────────────┘
```

规则：Master 先发包；Slave 必须 150μs 内回复；可多轮直到双方发空包；Slave 可借 Latency 跳过整个 Event。

## 二、150μs 回复窗口

BLE 极速反应的设计——快速关射频省电 + 避免信道长时间占用。Slave 不回则浪费这个 Event，下个 Interval 跳新信道。

## 三、自适应跳频（AFH）

### 跳频算法
`next_channel = (current + hopIncrement) % 37`，hopIncrement 须与 37 互质（5~16）。

### Channel Map 更新
37-bit 位图。某信道 CRC 连续失败 → Master 标记为坏 → 发 LL_CHANNEL_MAP_REQ 通知 Slave → 跳频时通过重映射算法自动绕过。

### 重映射规则
`remapping_index = hopIncrement % num_used_channels`，从可用信道列表重选。即使只剩 2 个可用信道仍能工作。

## 四、空包与 Slave Latency

**空包（Empty PDU）**：Header Length=0，表示"我还在但没数据"。

Latency 配合空包：Slave 每 Latency+1 个连接事件至少回复一次（带数据或空包），不能超过 Timeout/2，否则连接被误断。

## 五、连接参数动态更新

Slave 可发 LL_CONNECTION_PARAM_REQ 请求更新，Master 可发 LL_CONNECTION_UPDATE_REQ 单方面更新。Android 中 `requestConnectionPriority()` 就是发参数更新请求，但设备可拒绝。

## 六、Connection Event 结束条件

双方都发 MD=0（More Data=0）的包 → Master 决定关闭 → 或达到 CE Length 限制。

## 七、对开发的影响

- 150μs 回复由硬件控制，App 不可见
- 自适应跳频让连接很可靠，短暂干扰不会立刻断
- 参数更新只是"请求"，需用 `onConnectionUpdated()` 确认
- Wi-Fi 干扰导致连接变慢正常（可用信道减少），但通常不断
