## 一、BLE 是什么

BLE（Bluetooth Low Energy）是蓝牙 4.0 引入的低功耗分支，和经典蓝牙共享品牌但追求完全不同——为纽扣电池供电设备的间歇数据传输设计。

| 维度 | 经典蓝牙 | BLE |
|------|---------|-----|
| 目标 | 持续大数据流 | 间歇小数据，极低功耗 |
| 连接速度 | ~100ms | ~3ms |
| 速率 | 1~3 Mbps | 125 Kbps~2 Mbps |
| 信道 | 79 个 1MHz | 40 个 2MHz（3 广播 + 37 数据） |
| 典型场景 | 音频、文件传输 | 传感器、穿戴、信标、仪器仪表 |

## 二、协议栈三层架构

```
Application — 你的 App
    │
  HOST — GATT / ATT / GAP / SM / L2CAP  ← 主控芯片
    │
  HCI — 通信桥梁（UART）
    │
CONTROLLER — Link Layer / PHY  ← 蓝牙芯片
```

**物理归属**：手机上 Host 跑在主 CPU，Controller 跑在独立蓝牙 IC。IoT 设备（流量计）通常是单芯片 SoC，全部在一颗芯片上。

---

## 三、核心概念地图

```
Broadcaster ─→ 只广播不可连接（信标）
Observer    ─→ 只扫描不连接
Peripheral  ─→ 广播+可被连接（设备端，做Slave）
Central     ─→ 扫描+发起连接（手机端，做Master）

一次 BLE 交互：
  设备广播 → 手机扫描发现 → 连接 → 服务发现 → 读写数据 / 订阅通知
```

## 四、ATT 协议基础

**ATT（Attribute Protocol）= BLE 数据的"语言"，定义 Client 如何操作 Server 的数据。**

```
Client-Server 模型：
  Server（通常是 Peripheral/设备）— 拥有 Attribute Database
  Client（通常是 Central/手机）   — 发送 ATT 请求读写数据

Attribute = 数据库的一条记录：
  [Handle(2B)] [Type(UUID)] [Value(0~512B)] [Permissions(独立)]
  
  Handle = 主键（0x0001~0xFFFF）
  Type  = UUID（16-bit 标准或 128-bit 自定义）
  Permissions = 读/写各自独立设置：无需/需加密/需认证/需授权
```

## 五、ATT PDU 类型速查

### 读操作（5 种）

| PDU | 用途 |
|-----|------|
| Read Request/Response | 按 Handle 读单个值 |
| Read Blob Request/Response | 长数据分段读（Offset+Length） |
| Read By Type | 按 UUID 类型搜索 Characteristic |
| Read By Group Type | 按组类型搜索 Service |
| Find Information | 遍历所有 Attribute 的 Type（发现 Descriptor） |

### 写操作（4 种）

| PDU | 用途 | 有响应？ |
|-----|------|---------|
| Write Request/Response | 标准写 | ✅ |
| Write Command | 快速写（无确认） | ❌ |
| Signed Write Command | 带 CSRK 签名写（不加密链路） | ❌ |
| Prepare+Execute Write | 事务性长写（原子提交） | ✅ |

### 通知/指示（2 种）

| PDU | 可靠性 | 速度 | 适用 |
|-----|--------|------|------|
| Notification | 不可靠（丢了就丢了） | 快 | 周期性数据（流速/温度） |
| Indication+Confirmation | 可靠（需确认） | 慢 | 关键事件（报警/故障） |

### 错误响应

ATT_ERROR_RSP：Invalid Handle / Read/Write Not Permitted / Insufficient Authentication / Insufficient Encryption

## 六、GATT 数据组织

```
Profile（概念集合）
└── Service（功能模块，UUID 标识）
    ├── Include（可选，引用其他 Service）
    └── Characteristic（具体数据项）
        ├── Declaration [Properties][Value Handle][UUID]
        ├── Value（实际数据）
        └── Descriptor（附加配置）
            ├── CCCD (0x2902) — 通知/指示开关
            ├── CUD  (0x2901) — 用户可读名称
            └── Presentation Format (0x2904) — 数据格式
```

**Characteristic Properties**：Broadcast / Read / WriteWithoutResponse / Write / Notify / Indicate / SignedWrite / ExtProps

---

## 七、UUID 体系

```
16-bit UUID → 填入 Base UUID：0000XXXX-0000-1000-8000-00805F9B34FB

分配规律：
  0x18XX = Service      (0x180F=Battery, 0x180A=Device Info)
  0x2AXX = Characteristic (0x2A19=Battery Level)
  0x29XX = Descriptor    (0x2902=CCCD, 0x2901=CUD)

128-bit UUID = 厂商自定义（用 UUID v4 随机生成，不要手编）
```

## 八、服务发现流程

```
Round 1: READ_BY_GROUP_TYPE(0x2800) → 发现所有 Service（UUID + Handle 范围）
Round 2: READ_BY_TYPE(0x2803)      → 发现所有 Characteristic（Properties + ValueHandle + UUID）
Round 3: FIND_INFO                 → 发现所有 Descriptor（CCCD/CUD 等）

Android: discoverServices() → onServicesDiscovered()
```

## 九、读写与通知操作

| 操作 | Android API | 前置条件 |
|------|------------|---------|
| 读 | readCharacteristic() | discoverServices 完成 |
| 写（需确认） | writeCharacteristic() | 同上 |
| 写（无确认） | + WRITE_TYPE_NO_RESPONSE | 同上 |
| 通知/指示 | setCharacteristicNotification() + writeDescriptor(CCCD) | **必须写 CCCD！** |

**CCCD 是通知的开关**：
- `0x0000` = 关闭（默认）
- `0x0001` = 开启 Notification
- `0x0002` = 开启 Indication

**重连后必须重新写 CCCD**——连接断开时 CCCD 自动重置。

## 十、MTU 协商

```
默认 ATT MTU = 23 字节（有效数据仅 20 字节）
协商：ATT_EXCHANGE_MTU_REQ(512) → ATT_EXCHANGE_MTU_RSP(251) → MTU = min(512, 251) = 251

Android: requestMtu(512) → onMtuChanged()
关键：requestMtu 必须在 discoverServices 之前调用
```

## 十二、流量计 GATT 设计速记

```
Device Info (0x180A) — 标准 16-bit
Battery (0x180F) — 标准 16-bit
流量计测量服务 (128-bit 自定义)
  ├─ 瞬时流速 [Read|Notify] + CCCD
  ├─ 累计流量 [Read|Notify] + CCCD
  ├─ 采样间隔 [Read|Write]
  └─ 校准参数 [Read|Write, 需加密+认证]

安全分层：公开 Level1 / 业务 Level2 / 敏感 Level4
所有 Notify/Indicate 必须配 CCCD
标准 UUID 优先 → nRF Connect 等工具可自动解析
```
