# 6.6 读写与通知操作

## 一、Read 完整交互

readCharacteristic() → ATT_READ_REQ → 设备查 Handle→权限检查→返回 ATT_READ_RSP → onCharacteristicRead()

失败原因：距离干扰/权限不足/Handle 不存在

## 二、Write Request（带确认）

writeCharacteristic() → ATT_WRITE_REQ → ATT_WRITE_RSP → onCharacteristicWrite(SUCCESS)

**关键限制**：Android BLE 栈无内部队列，必须等上一个写回调完成后才能发下一个。

## 三、Write Command（无确认）

+ WRITE_TYPE_NO_RESPONSE → ATT_WRITE_CMD → 无响应 → 回调仅表示"已发出"，不保证到达

## 四、Notification（无确认推送）

**必须写 CCCD=0x0001**（通过 writeDescriptor：`ENABLE_NOTIFICATION_VALUE`）

数据来了 → ATT_HANDLE_VALUE_NOT → onCharacteristicChanged()

## 五、Indication（需确认推送）

CCCD=0x0002。收到后自动回 ATT_HANDLE_VALUE_CFM。Server 等 CFM 才发下一个，不能高频用。

## 六、流量计完整读写流程

```java
// 连接 → discoverServices → 找到 Characteristic
// 启用通知（必须！）
gatt.setCharacteristicNotification(flowChar, true);
cccd.setValue(ENABLE_NOTIFICATION_VALUE);
gatt.writeDescriptor(cccd);
// 等待 onCharacteristicChanged 回调
// 写配置
configChar.setValue(new byte[]{0x00, 0x0A});
gatt.writeCharacteristic(configChar);
```

## 七、常见坑

| 坑 | 原因 | 解决 |
|----|------|------|
| Notification 不回调 | 忘了写 CCCD | 必须 writeDescriptor(cccd) |
| 第二个写返回 false | 上一个写还没回调 | 串行：等回调再写下一个 |
| 重连后通知不来 | CCCD 在断连时归零！ | 每次重连必须重新写 CCCD |
| 读返回 GATT_FAILURE | 加密/权限问题 | 检查绑定状态 |

## ⚠️ 最重要一条

**重连后必须重新写 CCCD。** 连接断开时设备端 CCCD 自动重置为 0x0000。
