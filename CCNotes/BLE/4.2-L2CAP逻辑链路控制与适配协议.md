# 4.2 L2CAP — 逻辑链路控制与适配协议

## 一、L2CAP 解决两个问题

1. **协议复用**：Host 收到数据后给 ATT 还是 SM？靠 CID 区分
2. **分片重组（SAR）**：上层数据 > 链路层 Payload 时切分，接收方重组

## 二、CID：协议复用

| CID | 协议 |
|-----|------|
| 0x0004 | ATT (GATT 数据) |
| 0x0005 | L2CAP Signaling (参数协商) |
| 0x0006 | SM (安全管理器，配对/加密) |
| 0x0020~0x003F | LE Credit Based Channels (自定义信道, 4.2+) |

L2CAP 包格式：`[Length(2B)][CID(2B)][Payload(上层协议数据)]`

## 三、SAR：分片与重组

链路层 Payload 最大 27（旧）或 251（4.2+）字节。L2CAP 将大包切为多片，链路层用 LLID=10 标记起始片段、LLID=01 标记继续片段。接收方 L2CAP 重组后交上层。

## 四、L2CAP Signaling (CID=0x0005)

主要用于连接参数更新。Slave 通过 L2CAP 请求修改参数，Master 通过 LL Control PDU 修改。两条路都能到达。

## 五、LE Credit Based Flow Control (4.2+)

面向连接自定义信道的流控：Credit 机制类似 TCP 滑动窗口。ATT 信道 (0x0004) 不用（ATT 自带请求-响应流控）。主要用于 EATT (5.2+)。

## 六、逐层包裹实例

```
ATT Write Req: [Opcode][Handle][Value]
  → L2CAP: [Length][CID=0x0004][ATT包] (仅+4字节)
  → HCI ACL: [Handle][PB][L2CAP包]
  → LL Data: [Header][Payload][CRC]
  → PHY: GFSK → 无线电波
```

## 七、对开发的影响

- CID 透明，Android 协议栈自动处理
- SAR → MTU 小则吞吐低，阶段六讲 MTU 协商
- L2CAP 参数更新 → `onConnectionUpdated()` 可收到通知
- Credit 流控不影响 GATT 交互
