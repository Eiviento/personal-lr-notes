# 1.2 补充：协议栈各层的物理归属

> 提问：这些架构分别属于什么部分，是属于蓝牙模块还是属于开发板的主控，怎么区分各自的位置？

---

## 核心答案：取决于硬件架构，有两种典型方案

---

## 方案一：双芯片方案（手机、平板、早期设备）

```
┌────────────────────────────────┐          ┌──────────────────────┐
│     主控芯片 (主CPU/AP)         │  UART    │    蓝牙模块/芯片       │
│                                │←────────→│                      │
│  ┌──────────────────────────┐  │   HCI    │  ┌────────────────┐  │
│  │  Application (你的App)    │  │          │  │  SM ?          │  │
│  ├──────────────────────────┤  │          │  ├────────────────┤  │
│  │  GATT / GAP              │  │          │  │  Link Layer    │  │
│  ├──────────────────────────┤  │          │  ├────────────────┤  │
│  │  ATT                     │  │          │  │  PHY (射频)    │  │
│  ├──────────────────────────┤  │          │  └────────────────┘  │
│  │  SM                      │  │          │                      │
│  ├──────────────────────────┤  │          │     蓝牙芯片          │
│  │  L2CAP                   │  │          │   (如 Qualcomm CSR、  │
│  ├──────────────────────────┤  │          │    Broadcom BCM、     │
│  │  HCI Driver              │  │          │    Infineon CYW)      │
│  └──────────────────────────┘  │          └──────────────────────┘
│                                │
│  主控 = 应用处理器              │
│  (如 高通骁龙、联发科、A系列)    │
└────────────────────────────────┘
```

**物理归属**：
- **Host 协议栈（L2CAP + SM + ATT + GATT + GAP）** → 跑在**主控芯片**上（手机的主 CPU）
- **Controller（Link Layer + PHY）** → 跑在**蓝牙模块/芯片**上（独立的蓝牙 IC）
- **HCI** → 走物理 UART 线，连接两个芯片

**这就是你的 Android 手机的情况**：
- Android 系统 = Host，跑在主 CPU
- 蓝牙芯片 = Controller，独立的硬件，SoC 上总有一个蓝牙 IC
- 它们之间通过 UART 发 HCI Command/Event 通信

## 方案二：单芯片方案（IoT 设备的主流，你们的流量计大概率是这样）

```
┌──────────────────────────────────────────────────┐
│            单颗 BLE SoC（一片芯片搞定全部）        │
│                                                   │
│  ┌─────────────────────────────────────────────┐  │
│  │  ARM Cortex-M 核心 (MCU)                     │  │
│  │  ┌───────────────────────────────────────┐  │  │
│  │  │  Application (你的业务逻辑)             │  │  │
│  │  │  "流量 = 脉冲数 × 系数"                 │  │  │
│  │  ├───────────────────────────────────────┤  │  │
│  │  │  HOST 协议栈 (厂商 SDK 提供)            │  │  │
│  │  │  GATT/GAP/ATT/SM/L2CAP               │  │  │
│  │  ├───────────────────────────────────────┤  │  │
│  │  │  HCI 变成内部函数调用，不走物理总线      │  │  │
│  │  └───────────────────────────────────────┘  │  │
│  │                    ↕                        │  │
│  │  ┌───────────────────────────────────────┐  │  │
│  │  │  Radio 外设 (硬件)                      │  │  │
│  │  │  Link Layer + PHY                     │  │  │
│  │  │  2.4GHz 射频电路                       │  │  │
│  │  └───────────────────────────────────────┘  │  │
│  └─────────────────────────────────────────────┘  │
│                                                   │
│   常见型号: Nordic nRF52832/nRF52840             │
│            TI CC2541/CC2640/CC2652                │
│            Espressif ESP32-C3/ESP32-S3           │
│            Dialog DA14531/DA14683                │
│            Silicon Labs EFR32BG22                │
└──────────────────────────────────────────────────┘
```

**物理归属**：
- **全部都在同一块芯片上**！
- MCU 核心跑 Host + Application
- 同一个芯片的 Radio 硬件外设跑 Controller
- HCI 退化为内部 API 调用（比如 Nordic 的 SoftDevice 中的函数调用）

## 常见混淆场景辨析

### 场景 1：STM32 + 蓝牙透传模块（如 HC-05/HM-10）

```
┌──────────────┐    UART     ┌──────────────────┐
│  STM32 MCU   │←──────────→│  蓝牙透传模块     │
│  (主控开发板) │             │  (如 HM-10 模块)  │
│              │             │                   │
│  跑你的应用   │  发AT指令   │  内部是完整 SoC   │
│  解析流量数据 │  + 透传数据  │  (方案二的芯片)   │
│              │             │  所有协议栈都在   │
│              │             │  模块内部处理完   │
└──────────────┘             └──────────────────┘
```

**这种情况**：
- "蓝牙模块"（HM-10）内部其实是一颗完整的 BLE SoC（如 TI CC2541），它自己跑完所有协议栈
- STM32 通过串口跟模块通信，完全不碰 BLE 协议，只需要发 AT 指令和数据
- **优缺点**：省事但不可控，模块对所有手机都"看起来一样"，你无法定义自己的 Service/Characteristic（除非模块支持透传配置）

### 场景 2：直接在一颗 BLE SoC 上开发（不用外部 MCU）

```
┌────────────────────────────┐
│  Nordic nRF52832 开发板    │  ← 一块板子，一颗主芯片
│                            │
│  你的固件代码烧录在         │     整个 BLE 协议栈 + 应用逻辑
│  这颗芯片里：              │     都在这一颗芯片上运行
│  - 采集流量传感器脉冲      │
│  - 计算流量值              │
│  - 通过 BLE 上报到手机     │
└────────────────────────────┘
```

这通常是你们公司做流量计设备的方式——一颗 BLE SoC 搞定一切，不需要额外 MCU。

```
  主控芯片      蓝牙芯片           单颗 BLE SoC
 ┌────────┐   ┌────────┐       ┌────────────────┐

 │  Host  │   │ Contr- │       │  + Controller  │

 HCI走物理UART                  HCI是内部函数调用
```

## 对你开发的影响

| | 你在 Android 手机上开发 App | 公司同事在设备端开发固件 |
|---|---|---|
| **你面对** | Host 层及以上（Android BLE API） | 全栈（Nordic/TI SDK） |
| **你不用管** | Controller 和 HCI 的细节 | 但需要理解 |
| **你需要理解** | GATT/GAP/SM 的行为（因为 API 直接暴露这些概念） | 全都要懂 |

**关键是**：即便你只开发 Android 端，当出问题的时候（比如连接失败、配对失败、数据读不到），问题可能出在 Controller 层（射频干扰、跳频跟不上）、Host 层（协议栈 bug）、或者 App 层（API 用错了）。**所以每一层你都需要知道它在哪、干什么、出问题时怀疑谁。**

---

## 怎么判断一个硬件属于哪种方案？

| 判断方法 | 双芯片 | 单芯片 |
|---------|--------|--------|
| **看原理图** | 有独立的蓝牙 IC，和主控之间有一条 UART | 只有一颗主芯片，天线直接接到这颗芯片 |
| **看芯片型号** | 主控型号 ≠ 蓝牙芯片型号 | 一个型号搞定（如 nRF52、CC2640） |
| **看 SDK** | 通过 HCI 命令或 AT 指令控制蓝牙模块 | 直接调用 BLE API（如 Nordic SDK 的 `ble_nus_init()`） |
| **典型产品** | 手机、平板、树莓派+蓝牙 Dongle | 手环、传感器、信标、**流量计** |
