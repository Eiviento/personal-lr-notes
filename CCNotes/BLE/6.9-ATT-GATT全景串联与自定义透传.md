# 6.9 ATT-GATT 全景串联与自定义透传

> 本文为 6.1~6.8 的串联综述，把分散的知识点串成一条完整的"设备数据 → 手机 App"数据通路。
> 涵盖：GATT 表结构、ATT 报文格式、服务发现、读写通知操作、自定义透传服务设计。

---

## 一、ATT 和 GATT 的分工

```
ATT  = 动词 — 怎么读、怎么写、怎么推送（协议操作）
GATT = 名词 — 数据怎么组织（属性表树形结构）
```

类比：ATT = HTTP 的 GET/POST/PUT，GATT = REST API 的 URL 路由树。

---

## 二、GATT 表——设备里存的"数据 Excel"

设备的 Attribute Database 本质就是一张有序表，每个条目（Attribute）包含 Handle（编号）、UUID（类型）、Value（值）、Permissions（权限）：

```
Handle  UUID                  Value                  是什么
──────  ────────────────────  ─────────────────────  ────────────────
0x0001  00002800-...0000180F  0F18                    Service 声明: "电池服务"
0x0002  00002803-...00002A19  12 03 00 19 2A          Characteristic 声明
                                                        [Notify|Read][0x0003][0x2A19]
0x0003  00002A19-...          55                      Value: 电池 = 85%
0x0004  00002902-...          0000                    CCCD: 通知开关(关)

0x0010  00002800-...自定义128  12345678ABCD...         自定义 Service 声明
0x0011  00002803-...自定义128  12 12 00 ABCD...        自定义 Characteristic 声明
0x0012  自定义128-bit UUID     4048F5C3                Value: 流速 = 3.14(float)
0x0013  00002902-...          0001                    CCCD: 通知开关(开)
```

### 四级树形结构

```
Profile（概念层，不存在于 Attribute DB）
  └── Service（功能模块）
        │ Handle: Declaration (Type=0x2800, Value=Service的UUID)
        │
        └── Characteristic（具体数据项）
              │ Handle: Declaration (Type=0x2803, Value=[属性(1B)][Value的Handle(2B)][UUID])
              │ Handle: Value (Type=该特征UUID, Value=实际数据)
              │
              └── Descriptor（附加配置，如 CCCD）
                    Handle: CCCD (Type=0x2902, Value=0x0000/0x0001/0x0002)
```

### 关键类型 UUID

| UUID | 角色 | 说明 |
|------|------|------|
| 0x2800 | Service Declaration | 声明一个 Service 的开始和结束 Handle 范围 |
| 0x2801 | Secondary Service | 次要 Service |
| 0x2803 | Characteristic Declaration | 声明 Char 的属性、值所在 Handle、UUID |
| 0x2902 | CCCD | Client Characteristic Configuration Descriptor——Notify/Indicate 开关 |

## 三、ATT 报文——真实空中数据长什么样

### ATT PDU 通用格式

```
┌────────┬──────────────────────────────┐
│ Opcode │       Parameters             │
│ 1 字节  │   (0 ~ MTU-1 字节)           │
└────────┴──────────────────────────────┘

Opcode 结构:
  Bit [5:0] = Method (具体操作)
  Bit [6]   = Command Flag (0=需响应, 1=无需响应)
  Bit [7]   = Authentication Signature Flag (0=无签名, 1=带CSRK签名)
```

### 读操作（带确认，可靠）

```
手机: readCharacteristic(Handle=0x0003)
  → ATT_READ_REQ: [Opcode 0x0A] [Handle 0x0003]
  ← ATT_READ_RSP: [Opcode 0x0B] [Value 0x55]     ← 85%

大属性（超过 MTU）:
  → ATT_READ_REQ → 返回前 22 字节
  → ATT_READ_BLOB_REQ (Offset=22) → 下一页
  → 直到返回长度 < MTU-1 → 结束（由 Android 协议栈自动处理）
```

### 写操作

```
带确认（可靠，慢）:
  writeCharacteristic(Handle=0x0015, Value=0x000A)
    → ATT_WRITE_REQ: [Opcode 0x12] [Handle 0x0015] [Value 0x000A]
    ← ATT_WRITE_RSP: [Opcode 0x13]                  ← 只确认"收到了"

无确认（快，不保证到达）:
  writeCharacteristic(+ WRITE_TYPE_NO_RESPONSE)
    → ATT_WRITE_CMD: [Opcode 0x52] [Handle] [Value]
                          ↑ bit6=1 = Command 模式，设备不回任何东西
```

### 通知/指示（设备主动推数据）

```
启用通知（必须两步）:
  1. setCharacteristicNotification(char, true)
  2. writeDescriptor(CCCD, ENABLE_NOTIFICATION_VALUE = 0x0001)
     → ATT_WRITE_REQ: [Opcode 0x12] [CCCD Handle] [Value 0x0001]

Notify（无确认，快）:
  ← ATT_HANDLE_VALUE_NOT: [Opcode 0x1B] [Handle] [Value 3.14]
  ← ATT_HANDLE_VALUE_NOT: [Opcode 0x1B] [Handle] [Value 3.15]

Indicate（需确认，可靠但慢）:
  ← ATT_HANDLE_VALUE_IND: [Opcode 0x1D] [Handle] [Value]
  → ATT_HANDLE_VALUE_CFM: [Opcode 0x1E]              ← 手机自动回确认
```

| | Notify | Indicate |
|---|--------|---------|
| 确认？ | ❌ 不确认 | ✅ 设备等 CFM 后才发下一个 |
| 吞吐量 | 高（~3 倍于 Indicate） | 低 |
| 适用场景 | 高频数据（流速、温度） | 不可丢失事件（报警） |

## 四、服务发现——手机怎么读懂这张表

**手机首次连接时对 Attribute DB 一无所知。** Handle 是固件编译时分配的，每版本可能变——不能硬编码。必须通过 ATT 命令"探路"。

### 三大轮

```
Round 1: 发现所有 Service（找 0x2800 声明）
  → ATT_READ_BY_GROUP_TYPE_REQ (Start=0x0001, End=0xFFFF, Type=0x2800)
  ← 返回:
      0x0001~0x0004, UUID=0x180F (Battery Service)
      0x0005~0x0007, UUID=0x180A (Device Information)
      0x0010~0x0015, UUID=128-bit 自定义 (流量计 Service)

Round 2: 在每个 Service 范围内发现 Characteristic（找 0x2803 声明）
  → ATT_READ_BY_TYPE_REQ (Start, End, Type=0x2803)
  ← 返回每条: [Declaration Handle] [Properties] [Value Handle] [UUID]
      0x0002: [Notify|Read], Value@0x0003, UUID=0x2A19
      0x0011: [Notify|Read], Value@0x0012, UUID=自定义

Round 3: 扫描 Handle 间隙发现 Descriptor
  → ATT_FIND_INFO_REQ
  ← 返回: 0x0004: Type=0x2902 (CCCD), 0x0013: Type=0x2902 (CCCD)
```

三轮走完 → `onServicesDiscovered()` → `getServices()` 可用。

## 五、详解服务发现报文——以读 Service 列表为例

```
手机发送（LL Data PDU Payload）:

  L2CAP 头:
    Length = 3              ← Information Payload 长度
    CID    = 0x0004         ← 交给 ATT 层

  ATT PDU:
    10                      ← Opcode = READ_BY_GROUP_TYPE_REQ
    01 00                   ← Starting Handle = 0x0001
    FF FF                   ← Ending Handle = 0xFFFF（查全部）
    00 28                   ← 要找的 Type UUID = 0x2800

设备回复:
  ATT_READ_BY_GROUP_TYPE_RSP:
    Length = 6              ← 每条记录 6 字节
    01 00                   ← Handle 范围起始 = 0x0001
    04 00                   ← Handle 范围结束 = 0x0004
    0F 18                   ← Service UUID = 0x180F (Battery)
    
    05 00 07 00 0A 18       ← Service 2: 0x0005~0x0007, UUID=0x180A
    
    10 00 15 00             ← Service 3: 0x0010~0x0015,
    [16 字节自定义 UUID]      ← 自定义 Service
```

## 六、UUID——为什么知道 0x180F 是电池、0x180A 是设备信息

**蓝牙 SIG 维护了一本"标准字典"**——给常用功能分配了短 UUID：

| 16-bit UUID | 含义 | 类型 |
|------------|------|------|
| 0x180F | Battery Service | Service |
| 0x180A | Device Information | Service |
| 0x2A19 | Battery Level | Characteristic |
| 0x2A29 | Manufacturer Name | Characteristic |
| 0x2902 | CCCD | Descriptor |

16-bit UUID 是 128-bit 的缩写——填入标准 Base UUID：
```
Base UUID: 0000XXXX-0000-1000-8000-00805F9B34FB
           ↑ 16-bit 填入这里

0x180F → 0000180F-0000-1000-8000-00805F9B34FB
```

**手机 App / nRF Connect 的代码里硬编码了这本字典**——读到 0x180F 就显示"电池服务"和电池图标，读到自定义 128-bit UUID 则显示原始 Hex 字符串。

### 自定义 128-bit UUID

用 UUID v4 随机生成（`uuidgen` 或 `[Guid]::NewGuid()`），和你的固件约定好。只有你自己的 App 知道这个 UUID 后面的数据格式、含义、单位：

```java
// 你的 App:
if (charUuid.equals(MY_FLOW_RATE_UUID)) {
    name = "瞬时流速";
    format = FORMAT_FLOAT;
    unit = "m³/s";
}
```

## 七、自定义透传服务设计

**ATT/GATT 只负责把字节送过去。** 字节里的命令号、格式、含义——全是你和固件约定的。这就是你自己的应用层协议。

### 经典双 Characteristic 透传模型

```
自定义透传 Service (128-bit UUID):

  └── TX Characteristic (设备→手机)
        Properties: Notify
        手机写 CCCD 开启通知
        Value 格式: [命令号(1B)][数据长度(1B)][Payload(变长)]

  └── RX Characteristic (手机→设备)
        Properties: Write / Write Without Response
        手机直接写 Value
        Value 格式: [命令号(1B)][数据长度(1B)][Payload(变长)]
```

### 协议实例

```
手机→设备（写 RX）:
  0x01 0x04 0x00 0x00 0x00 0x0A → 命令号=0x01(设置采样间隔), 长度=4, 数据=10秒
  0x02 0x01 0x00                → 命令号=0x02(立即读流速), 长度=1, 无参数
  0x03 0x00                     → 命令号=0x03(校准归零), 长度=0, 无需参数

设备→手机（TX 推送）:
  0x81 0x04 0x40 0x48 0xF5 0xC3 → 命令号=0x81(流速数据), 长度=4, 数据=3.14(float)
  0x82 0x02 0x00 0x55           → 命令号=0x82(电量), 长度=2, 数据=85%
  0xFF 0x01 0x01                → 命令号=0xFF(报警), 长度=1, 数据=1(流量过大)
```

**长度字段（第二个字节）的作用**：告诉接收方 Payload 有多少字节。没有它，接收方不知道在哪停——可能把下一条命令的数据当成当前命令的参数。这是所有二进制协议 TLV 的基础，和 AD Structure LTV 同一个道理。

### 自定义透传命令模式

```
零参数命令（归零、读数触发）:
  [Cmd][Len=0x00]  → 共 2 字节

简单参数命令（设间隔、阈值）:
  [Cmd][Len=0x04][4字节值]  → 共 6 字节

批量数据（固件升级、历史记录）:
  [Cmd][Len=0xF6][246字节数据]  → 共 248 字节
  超过 MTU-5 须自己分片
```

## 八、数据长度限制（MTU 约束）

**你的 Payload 受到 ATT MTU 的三层约束：**

```
ATT 包总长 ≤ MTU

ATT 包 = [Opcode(1B)] [Handle(2B)] [你的 Cmd+Len+Payload]
          ───────────── ATT 头(3B) ─────────────

最大 Payload = MTU - 3(ATT头) - 2(Cmd+Len)
             = MTU - 5
```

| MTU | 你的 Payload 最大 | 场景 |
|-----|-----------------|------|
| 23（默认） | 18 字节 | 不协商，够传小命令 |
| 251（协商后） | 246 字节 | 常见设备，够传绝大多数数据 |
| 517（Android 极限） | 512 字节 | 够传大块数据 |

**建议**: `onConnectionStateChange(CONNECTED)` → `gatt.requestMtu(512)` → `onMtuChanged(SUCCESS)` → `discoverServices()`

## 九、读写回复机制——ATT 各操作的回复不同

| ATT 操作 | 有回复？ | 回复里带数据？ | 说明 |
|----------|---------|-------------|------|
| ATT_READ_REQ | ✅ | ✅ 带 Value | 读到的数据在 Response 里 |
| ATT_WRITE_REQ | ✅ | ❌ 空确认 | 只说"收到了"，不是执行结果 |
| ATT_WRITE_CMD | ❌ | — | 无任何回复 |
| ATT_HANDLE_VALUE_NOT | ❌ | — | 设备推送后不等确认 |
| ATT_HANDLE_VALUE_IND | ✅ | ❌ 空确认 | 手机自动回 CFM，仅表示"收到了" |

### 自定义透传中的"命令-回复"完整流程

```
同步模式（简单，两次往返）:
  1. 写 RX Char [Cmd=读流速]
     → ATT_WRITE_REQ → ATT_WRITE_RSP(仅确认收到)
  2. 稍等 → 读 TX/Result Char
     → ATT_READ_REQ → ATT_READ_RSP(带值!)  ← 返回值直接在 Response 里

异步模式（主流，不阻塞）:
  1. 写 RX Char [Cmd=读流速]
     → ATT_WRITE_REQ → ATT_WRITE_RSP(仅确认消息到达)
  2. 设备处理 → 推送 Notification
     → onCharacteristicChanged(数据)  ← 这才是命令执行结果

异步是 BLE 主流——Write Send 走命令，Notification 走回复，互不阻塞
```

## 十、完整数据通路（从开机到读到第一个值）

```
1. 连接建立（见 3.5）
   ══════ Connection ══════

2. requestMtu(512) → MTU 协商

3. discoverServices()
   三轮探测 → GATT 表映射完成
   onServicesDiscovered()

4. 找到 Characteristic
   service.getCharacteristic(MY_FLOW_RATE_UUID)

5. 开启通知（Critical!）
   gatt.setCharacteristicNotification(flowChar, true)
   writeDescriptor(cccd, 0x0001)  → 激活 CCCD

6. 等数据
   onCharacteristicChanged(data) → updateUI(...)

7. 写命令（可选）
   写 RX Char [Cmd=0x01 设采样间隔=10s]
   onCharacteristicWrite(SUCCESS)

8. ⚠️ 断开 → CCCD 自动归零！
   每次重连必须重新: setCharacteristicNotification + writeDescriptor(CCCD)
```

## 十一、GATT/ATT 分层全貌

```
                        App (你的代码)
                          │
                     readCharacteristic()
                     writeCharacteristic()
                     onCharacteristicChanged()
                          │
              ┌───────────┴───────────┐
              │       GATT 层         │  ← "名词"
              │   getService(UUID)    │     在设备表里找东西
              │   getCharacteristic() │
              └───────────┬───────────┘
                          │
              ┌───────────┴───────────┐
              │       ATT 层          │  ← "动词"
              │   ATT_READ_REQ/RSP    │     实际发什么包、等什么回复
              │   ATT_WRITE_REQ/RSP   │
              │   ATT_NOT/IND         │
              └───────────┬───────────┘
                          │
              ┌───────────┴───────────┐
              │   L2CAP (CID=0x0004)  │  ← 加上 CID，告诉对面交给 ATT
              └───────────┬───────────┘
                          │
                      LL Data PDU
                      → PHY → 空中
                          
              ──────────────────────────
                          
                 设备端: 同样的层次反向解包
                 LL → L2CAP(CID=0x0004) → ATT → 查 GATT 表 → 读写 Flash → 返回
```

## 十二、核心踩坑速查

| 坑 | 原因 | 解决 |
|----|------|------|
| Notification 不回调 | **忘了写 CCCD** | `writeDescriptor(cccd, 0x0001)` |
| 重连后不通知 | **CCCD 断连自动归零** | 每次重连必须重新写 CCCD |
| 第二个写返回 false | **Android BLE 无内部队列** | 等上一个写回调完成再发下一个 |
| discoverServices 慢 | 100ms Interval × 大量往返 | 固件 Service/Char 控制在最小够用 |
| 硬编码 Handle | **固件升级 Handle 会变** | 永远用 UUID 做服务发现 |
| 单次写超 MTU-3 | 超出 ATT 包容量 | requestMtu(512) 或自己分片 |

## 问答记录

### Q1: 设备里到底存的什么数据？实际拿到的报文是什么样的？

设备 Flash 里是一张有序属性表（Handle | UUID | Value | Permissions），像一张 Excel 表。手机通过 ATT_READ_BY_GROUP_TYPE_REQ、ATT_READ_BY_TYPE_REQ、ATT_READ_REQ 等命令逐层探测和读取，每个包都是带 Opcode + Handle + 参数的二进制格式。空中传输走 LL Data PDU → L2CAP(CID=0x0004) → ATT PDU。

### Q2: 为什么知道 0x180F 是电池、0x180A 是设备信息？自定义 UUID 是什么意思？

蓝牙 SIG 维护标准 UUID 字典——16-bit UUID `0x180F` 填入固定 Base UUID 变成 `0000180F-0000-1000-8000-00805F9B34FB`，规范定义它代表 Battery Service。手机/工具代码里硬编码了这本字典。自定义 128-bit UUID 是你用 UUID v4 随机生成的，只有你自己的 App 和固件约定好它的含义。

### Q3: 自定义透传服务是怎么样的？

在设备上建一个自定义 Service（128-bit UUID），包含 TX（Notify，设备→手机）和 RX（Write，手机→设备）两个 Characteristic。手机写 RX 发命令，设备通过 TX 推送回复。ATT/GATT 只管把字节送过去，字节里的命令号、格式、含义是你的应用层协议——就像 HTTP 送 JSON，JSON 内容是你的业务逻辑。

### Q4: 命令协议中 `0x01 0x04` 后面的 `0x04` 是什么意思？

数据长度——告诉接收方 Payload 有多少字节。`0x01 0x04 0x00...` = 命令号 0x01，长度 4 字节，Payload = 4 字节的采样间隔值。没有长度字段，接收方不知道读几个字节才读完一条命令——可能把下一条命令的数据当参数吃掉。

### Q5: 自定义协议能发多少数据？

受 ATT MTU 约束：`最大 Payload = MTU - 3(ATT头) - 2(Cmd+Len) = MTU - 5`。默认 MTU=23 → 只能传 18 字节；协商 MTU=251 → 可传 246 字节；Android 极限 MTU=517 → 可传 512 字节。超过需自己分片。

### Q6: 主机对服务读写能获取回复值吗？

取决于 ATT 操作类型。ATT_READ_REQ → Response 直接带值。ATT_WRITE_REQ → Response 仅确认"收到了"，不带执行结果。自定义透传中的命令结果靠 TX Characteristic 的 Notification 异步返回——Write 走命令，Notify 走回复，互不阻塞。

### Q7: ATT Opcode bit 7 签名标志（CSRK）是什么？

不同于链路层 AES 加密——签名是给 ATT 数据贴防伪标签，数据本身不加密。用于数据本身不敏感但需要防篡改的场景（如医疗传感器数据）。CSRK 在 SMP Phase 4 和 LTK/IRK 一起分发。
