# 3.7 链路层 PDU 格式

## 一、PDU 总分类

```
链路层 PDU（连接状态）
├── LL Data PDU     ← 承载上层数据 (L2CAP→ATT→GATT)
└── LL Control PDU  ← 链路层自身管理
```

## 二、LL Data PDU 结构

```
Header(2) + Payload(0~251B) + MIC(4B, 加密时) + CRC(3B)
```

**Header 字段**：

| 字段 | 大小 | 含义 |
|------|------|------|
| LLID | 2 bit | 01=数据空包/继续片段, 10=数据起始片段, 11=控制PDU |
| NESN | 1 bit | Next Expected SN（确认+期望对方下一个包的序列号） |
| SN | 1 bit | Sequence Number（本包序列号, 0/1翻转，相同=重传） |
| MD | 1 bit | More Data（1=连接事件内还有更多数据） |
| RFU | 2 bit | Reserved |
| Length | 8 bit | Payload 长度，最大 251（BLE 4.2+, 之前27） |

**NESN+SN 确认重传机制**：SN 0/1 翻转，连续两包 SN 相同说明是重传；NESN 告诉对方"你上一包我收到了，期待你的下一个 SN"。

## 三、LL Control PDU 结构

```
Header(LLID=11) + Opcode(1B) + Control Data(可变)
```

| PDU | Opcode | 方向 | 作用 |
|-----|--------|------|------|
| LL_CONNECTION_UPDATE_REQ | 0x00 | M→S | 更新连接参数 |
| LL_CHANNEL_MAP_REQ | 0x01 | M→S | 更新信道图 |
| LL_TERMINATE_IND | 0x02 | 双向 | 断开连接 |
| LL_ENC_REQ/RSP | 0x03/04 | M↔S | 加密请求/响应 |
| LL_START_ENC_REQ/RSP | 0x05/06 | M↔S | 开始加密 |
| LL_FEATURE_REQ/RSP | 0x08/09 | 双向 | 特性交换 |
| LL_VERSION_IND | 0x0C | 双向 | 蓝牙版本号 |
| LL_REJECT_IND | 0x0D | 双向 | 拒绝不支持的 Control |
| LL_PHY_REQ/RSP | 0x14/15 | 双向 | PHY 切换 (5.0) |

## 四、加密范围

- **Header**：始终明文（方便链路层快速解析）
- **Payload + MIC**：加密（保护 GATT 数据 + 防篡改）
- **CRC**：始终明文（物理层误码检测，和加密无关）

## 五、数据包逐层封装实例

```
App: readCharacteristic() → ATT_READ_REQ(3B)
  → L2CAP 加 Header: [Length][CID=0x0004][ATT_READ_REQ] (7B)
  → Link Layer: [Header(LLID=10,SN,NESN,MD)][L2CAP包][CRC]
  → PHY: GFSK 调制 → 2.4GHz 发出
```

## 六、阶段三知识地图

广播阶段：状态机→广播信道→广播PDU(4种)→广播参数
         ↓ SCAN_REQ/RSP + CONNECT_IND
连接阶段：状态机(Connection)→数据信道跳频→LL Data/Control PDU→连接参数(Interval/Latency/Timeout/Channel Map)

## 问答记录

### Q1：LL Data PDU 和 LL Control PDU 是什么关系？

**核心比喻**：

```
LL Data PDU    → 运货的卡车（ATT/GATT 业务数据）
LL Control PDU → 修路的工程车（管理连接本身）
```

它们共用同一个物理信道、同一种包格式、同一套 SN/NESN 确认机制，**区别仅在于 Header 里的 LLID 字段**：

| LLID | 类型 | 含义 |
|------|------|------|
| 10 | LL Data PDU | 数据起始片段，Payload 传给 L2CAP |
| 01 | LL Data PDU | 空包（维持心跳）或分片续传 |
| 11 | LL Control PDU | 控制命令，链路层自己处理 |

---

**同一条连接事件里，Control 和 Data 可以交替发**：

```
连接事件（每 30ms 一次）

  ┌─── Master ──────────────────────────────────→ Slave ───┐
  │                                                         │
  │ ① LL Control PDU (LLID=11):                            │
  │    [Header|Opcode=0x03|ControlData]                      │
  │    LL_ENC_REQ → "先加密这条链路"                          │
  │    ← Opcode 在 LL 层，明文                                │
  │                              ← LL_ENC_RSP                │
  │                                                         │
  │ ② LL Control PDU (LLID=11):                            │
  │    [Header|Opcode=0x05|ControlData]                      │
  │    LL_START_ENC_REQ → "开始用新密钥"                      │
  │    ← Opcode 在 LL 层，明文                                │
  │                              ← LL_START_ENC_RSP          │
  │                                                         │
  │ ③ LL Data PDU (LLID=10):                               │
  │    [Header|──── Payload(加密) ────|MIC|CRC]              │
  │    ← LL 层没有 Opcode 字段！                              │
  │    解密后 Payload 里 → L2CAP → ATT [Opcode=0x12]         │
  │                              ← LL Data PDU(空包)          │
  └─────────────────────────────────────────────────────────┘
```

**比较**：

| | LL Data PDU | LL Control PDU |
|---|---|---|
| **识别** | LLID = 10 或 01 | LLID = 11 |
| **谁处理** | 扔给 L2CAP → ATT → App | 链路层自己处理，不上传 |
| **内容** | GATT 读写/通知数据 | 加密协商、参数更新、断开 |
| **有 Opcode 吗** | LL 层没有；ATT 层的 Opcode 在加密 Payload 里 | LL 层有 1 字节 Opcode，始终明文 |
| **对 App 可见** | ✅ onCharacteristicRead/Write/Changed | ❌ App 看不到 Control PDU |

---

### Q2：开始加密后的 LL Data PDU 中就没有 Opcode 了吗？

**LL Control PDU 有 LL 层的 Opcode（始终明文），LL Data PDU 没有 LL 层 Opcode，但它肚子里装的 ATT 包有 ATT 层的 Opcode（加密后从空中不可见，解密后完好无损）。**

这是两个层级的两套 Opcode：

| | LL Control PDU Opcode | ATT PDU Opcode |
|---|---|---|
| **所在层** | 链路层 | ATT 层（在 Data PDU 加密 Payload 里） |
| **位置** | LL Header 后的第 1 字节 | L2CAP Payload 的第 1 字节 |
| **例子** | 0x03=LL_ENC_REQ, 0x02=LL_TERMINATE_IND | 0x0A=ATT_READ_REQ, 0x12=ATT_WRITE_REQ |
| **加密影响** | 永远不加密，始终明文 | 加密后随 Payload 一起变成密文 |

**加密不影响协议结构本身**——ATT Opcode 一直在那儿，解密后还是原来的值。

---

**LL Data PDU 完整结构示例（发送 ATT Write Request to CCCD）**：

```
┌──────────────┬────────────────────────────────────────────┬───────┬─────┐
│  LL Header   │              Payload (加密)                 │  MIC  │ CRC │
│  LLID=10     │ ┌──────────┬───────────┬─────────────────┐ │       │     │
│  SN=0,NESN=0 │ │L2CAP头   │ ATT Opcode│  ATT Data       │ │       │     │
│  MD=0,Len=7  │ │Len=3     │  0x12     │ (Handle+CCCD值) │ │       │     │
│  (明文)       │ │CID=0x0004│(WRITE_REQ)│                 │ │(密文) │(明文)│
└──────────────┴─┴──────────┴───────────┴─────────────────┴─┴───────┴─────┘
                  └──────────────── 全部被 AES-CCM 加密 ────────────┘
```

**Header 永远明文**（链路层需要快速解析 SN/NESN/Length 做确认和重传），**Payload + MIC 被 AES-CCM 加密**，**CRC 永远明文**（物理层误码校验，和加密无关）。

### Q3：MIC 是什么？

**MIC = Message Integrity Check（消息完整性校验）**——加密包的"防伪印章"，4 字节，用于验证数据**没有被篡改**且**确实来自配对设备**。

**生成方式**：AES-CBC-MAC(SessionKey, Payload) → 4 字节 MIC，然后 MIC 随 Payload 一起被加密。

**校验方式**：接收方解密后重新计算 Payload 的 MIC，与收到的 MIC 比对——一致则通过，不一致则丢弃。

**MIC vs CRC**：

| | MIC | CRC |
|---|---|---|
| **目的** | 防**人为篡改** | 防**物理误码**（噪声） |
| **生成方式** | AES-CMAC（用 Session Key） | 多项式除法（公开算法） |
| **谁能验证** | 只有持有 LTK 的配对双方 | 任何人都能算 |
| **明文/密文** | 密文（随 Payload 加密） | 始终明文 |
| **被破坏后果** | 包丢弃，连接可能断开 | 包丢弃，等待重传 |

> CRC 防天灾，MIC 防人祸。

对于流量计场景：如果攻击者在中途把流速值从 0x01 改成 0xFF——有 MIC 则帧被丢弃等下一帧正确数据，无 MIC 则 App 显示错误读数。

### Q4：底层都是广播，蓝牙协议交互就是在不停地广播对吗？

**对了一半。** 物理层确实如此——所有 BLE 通信都是 2.4GHz 电磁波辐射，无方向性。但链路层通过三种机制在广播的物理介质上构建了排他的逻辑信道：

```
                    PHY 层              Link Layer
                   ────────           ────────────
广告阶段：    电磁波广播 ←──────→  真广播：谁都能解析
                              (固定 37/38/39 信道，固定格式)

连接阶段：    电磁波广播 ←──────→  假广播：只有配对设备能解析
                              (跳频+时间同步+Access Address 过滤)
```

**三种排他机制**：

1. **Access Address（32 位随机数）**：连接建立时 CONNECT_IND 下发，附近设备即使碰巧在同一信道，Access Address 不匹配，链路层直接丢弃
2. **自适应跳频**：37 个数据信道之间跳跃，外人不知道下一跳去哪个信道
3. **精确时间同步**：双方只在约定时刻（连接事件）醒来几百 μs，其余时间深度睡眠

> 像两个人约好：周三下午 3 点 17 分，在 3 号楼的第 18 个房间见面。别人既不知道时间、不知道地点，到了门口也没暗号（Access Address）。

这也是 **BLE 低功耗的关键**——不是"一直广播"，而是"精准约会、约完就睡"：

```
连接事件（30ms 间隔）：

  30ms ──────────────────────────────── 30ms ──────────────────
  ┌────┐                              ┌────┐
  │发包│  ← 只在这 300μs 醒着            │发包│
  └────┘                              └────┘
        └──────── 29.7ms 深度睡眠 ────────┘
```

> 无线电层面确实都是广播（电磁波无方向性），但链路层通过 Access Address + 自适应跳频 + 精确时间同步三种机制，在广播的物理介质上构建了一个排他的逻辑信道。**连接不是"不停广播"，而是"不停约会"。**

### Q5：为什么 LL Data PDU 中不包含 HCI ACL 包头？

**HCI 头不过空中那道线。** HCI 是 Host 和 Controller 两片芯片之间的内部通信协议，不是空中协议的一部分。

```
你的手机：

  ┌──── Host（主 CPU）────┐          ┌── Controller（蓝牙 IC）──┐
  │ App → ATT → GATT      │  HCI ACL │ 链路层 → PHY            │
  │          ↓            │ ──────→  │   ↓                     │
  │        L2CAP          │  UART    │  组装 LL Data PDU       │
  │     [包好 ATT 数据]    │          │  [拆掉 HCI 头]           │
  └───────────────────────┘          └─────────────────────────┘
```

**HCI 像快递面单**——Host 发给 Controller 时用来标注"这是发给哪条连接的"，Controller 收到后撕掉面单，只把 L2CAP 包装到 LL Data PDU 里。

完整的数据封装-解封链路：

```
发送端：ATT → L2CAP → +HCI头(发往Controller) → =Controller拆HCI头= → LL Data → PHY → 空中
接收端：空中 → PHY → LL Data → +HCI头(发往Host) → =Host拆HCI头= → L2CAP → ATT → App

HCI 头只在芯片间 UART 上传，永远不进入 2.4GHz 频段
```
