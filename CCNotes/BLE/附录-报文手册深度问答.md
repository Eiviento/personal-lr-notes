## Q1: 广播连接阶段，主机发 CONNECT_IND 后，从机不用回复吗？直接从广播态变为已连接？

**是的，从机收到 CONNECT_IND 后不需要回复任何 ACK，直接从 Advertising 态切到 Connection 态。**

CONNECT_IND 本身携带了连接所需的全部 36 字节参数（Access Address、CRC Init、Interval、Latency、Timeout、Channel Map、Hop Increment、WinOffset 等），双方拿到后就有了一致的"约定"——什么时候第一次见面（WinOffset）、用什么加密参数（CRC Init）、在哪些信道跳（Channel Map + Hop Increment）。不需要再握手确认。

连接的真正确认发生在**第一次连接事件**：Master 按约定时刻发包，Slave 必须在 150μs 内回复。如果 Slave 回复了 → 连接正式确立；如果没回复 → 不是立刻断开，而是等 Supervision Timeout 耗尽后才判失败。

**关键约束**：CONNECT_IND 必须在同一广播事件内发出——设备广播后的监听窗口仅百微秒级。

---

## Q2: 主机发 CONNECT_IND 后，第一次连接事件从机没回复会怎样？是一次失败就断开吗？

**不是。单个连接事件失败 ≠ 断开。只有连续超过 Supervision Timeout 没有有效回复才会断开。**

```
CONNECT_IND 发出
     │
     ├─ 等待 WinOffset
     ▼
第一次连接事件 Anchor Point ←── Master 发包
     │
     ├─ 从机回了 ──→ ✅ 正常，后续按 Interval 继续
     ├─ 从机没回 ──→ ❌ 这个事件空了，但不判失败，等下一个 Interval 再试
     │
     ▼
 Supervision Timeout 到期 ──→ 💔 宣布断开
```

Slave Latency 机制本身就允许从机跳过若干次不回复（约束：`(1+Latency) × Interval × 2 ≤ Timeout`），偶尔丢包或从机睡着不回复是正常的。

**边界情况**：如果从机压根没收到 CONNECT_IND（干扰丢包），从机继续广播，Master 却已进入 Connection 态在等。这种情况 Master 会在 Timeout 后断开回到 Standby；从机什么都不知道，继续广播——两边各走各的。这是一些 BLE "假连上" 的原因。

## Q3: 定向可连接广播（ADV_DIRECT_IND）的 6 字节 InitA 是白名单吗？白名单最大限制是多少？

**ADV_DIRECT_IND 的 6 字节 InitA 不是白名单，是单一目标地址。** 它只能填一个设备的 MAC 地址，意思是"只有这一个设备可以连我"。

真正的白名单（White List）是链路层的一个独立功能，存在控制器内存里，可以存储多个地址。配合 ADV_IND 使用：广播人人可见，但只有白名单内的设备发来的 CONNECT_IND 才会被接受。

**白名单容量**：BLE 规范没有规定硬上限，由芯片厂商实现决定。可通过 HCI 命令 `LE Read White List Size` 查询。

| 芯片 | 白名单容量 |
|------|-----------|
| Nordic nRF51 | 8 |
| Nordic nRF52/53 | 8 |
| TI CC254x | ~10 |
| TI CC26xx | ~10 |
| Dialog DA145xx | 8 |
| Silicon Labs EFR32 | 8~16 |

**典型值 8 个。**

| | ADV_DIRECT_IND | 白名单 + ADV_IND |
|---|---|---|
| 能存储的地址数 | **1 个** | 8~16 个 |
| 带广播数据？ | ❌ 不能 | ✅ 31 字节 AdvData 照常 |
| 非目标设备看得到？ | ❌ 链路层直接丢弃 | ✅ 能看到但连不上 |

## Q4: 设备发定向可连接广播时，怎么知道 InitA 要填谁的地址？

**InitA 来自绑定表，绑定表来自第一次配对。**

```
首次连接（ADV_IND 广播，人人可发现）
     │
     ▼
配对 + 绑定
     │  配对阶段，Central 的身份地址（Public 或 Static Random）
     │  会交换给 Peripheral
     ▼
Peripheral 把地址存入 Flash 绑定表：
     ├── Peer Identity Address: 0xAA:BB:CC:DD:EE:FF
     ├── LTK
     ├── EDIV / Rand
     └── IRK
     │
     ▼
断开 → 下次想重连 → 从绑定表读地址 → 填进 InitA → 发 ADV_DIRECT_IND
```

**RPA（Resolvable Private Address）的问题**：现代手机默认每 15 分钟换一次 RPA。绑定表存的是 Identity Address（根地址），手机收到定向广播后能用 IRK 认出自己的身份地址并响应——所以定向广播仍然有效。

**实际工程中的重连策略**：
1. 先发 ADV_DIRECT_IND，InitA 填存储的身份地址
2. 如果 Timeout（1.28s）没连上 → 切回 ADV_IND（通用广播）
3. 手机扫描到 ADV_IND → 发起连接 → 链路层用 IRK 认出对方 → 跳过配对直接加密

## Q5: LL_CONNECTION_UPDATE_REQ 是 Control PDU，没有 Response。如果从机没收到，主机会直接切到新参数吗？

> 注：手册中记为 `LL_CONNECTION_UPDATE_REQ`。在 BLE 核心规范中，Opcode 0x00 的正式名称是 `LL_CONNECTION_UPDATE_IND`（Indication，单方面指示，无 Req/Rsp 配对）。本文沿用规范名称。

**是的，Master 在 Instant 时刻一到就切到新参数，无论 Slave 有没有收到。如果 Slave 没收到 → 两边参数分叉 → 彻底失联 → Timeout 断开。**

```
事件 N：Master 发 LL_CONNECTION_UPDATE_IND (Instant=N+6)
         包丢了，Slave 没收到

事件 N+1 ~ N+5：一切照旧，双方还在老参数上通信，Master 等着 Instant

          ═══════ Instant (N+6) ═══════

Master 侧：已切到新参数               Slave 侧：还在老参数
  ● Interval = 30ms                    ● Interval = 50ms
  ● 信道 = (ch + hop_new) % 37         ● 信道 = (ch + hop_old) % 37

          ↓↓ 信道不同 + 时间错位 → 彻底失联 ↓↓
          ↓↓ Supervision Timeout 到期 → 双方各自断开
```

**但实际上被两层机制防住了**：

### 第一层：链路层 ACK（最可靠）

LL_CONNECTION_UPDATE_IND 语义上不需要 Response，但它作为数据包，底层仍然走链路层 ACK：

```
Master 发数据包（Payload = LL_CONNECTION_UPDATE_IND）
          ↓
Slave 收到 → 150μs 内回 ACK
          ↓
Master 收到 ACK → ✅ 确信对方收到了，Instant 后放心切
Master 没收到 ACK → ❌ 重传，直到 Instant 前
Instant 前都没传成功 → Master 放弃更新
```

### 第二层：Instant 延迟缓冲

Instant 通常留 6~32 个事件的窗口，给足了重传时间。

**真正会翻车的场景**：ACK 回了但 Instant 前 Slave 意外重启、Instant 前后信道剧烈恶化、极端时钟漂移。结果都是 Timeout 断开，不会在假连接上鬼打墙。

## Q6: 不管是否经过确认（Just Works vs Numeric Comparison），两个设备之间传输的数据都是经过加密的吗？

**是的，数据都经过 AES-CCM 加密。但"加密了"不等于"安全"。**

| 场景 | 加密了？ | 密钥有多强？ | 能防被动窃听？ | 能防主动 MITM？ |
|------|---------|-------------|--------------|----------------|
| SC + Numeric Comparison | ✅ | 128-bit | ✅ | ✅ |
| SC + Just Works | ✅ | 128-bit | ✅ | ❌ **MITM 能解密** |
| Legacy + Passkey | ✅ | ~20-bit | ⚠️ 可暴力破解 | ⚠️ |
| Legacy + Just Works | ✅ | TK=0, 形同虚设 | ❌ 旁听者也能算 STK | ❌ |

**SC Just Works 下的 MITM 原理**：

```
攻击者 M 拦截配对双方的 ECDH 公钥，替换成自己的：
  Phone ↔ M: SessionKey₁ → M 能解密、修改、转发
  M ↔ 耳机:  SessionKey₂ → M 能解密、修改、转发

M 完全透明地读、改、转发所有数据。
加密是真实的(AES-CCM, 128-bit)，但 M 拥有密钥。
```

> **SC Just Works：加密了但不知道跟谁加密。Numeric Comparison：加密了且确认了对方身份。Legacy Just Works：加密了但任何人旁听都能算出来密钥。**

加密和认证是两个独立属性——加密保数据机密性，认证保通信方身份。BLE 的 AES-CCM 加密始终生效，认证强度取决于配对方式。

## Q7: 加密过程涉及广播包吗？

**不涉及。** 加密的整个协商过程全部发生在连接建立之后，走 LL Data PDU：

```
LL_ENC_REQ/RSP → LL_START_ENC_REQ/RSP → AES-CCM 加密通信
    ↑                    ↑
   全部走 LL Data PDU，在 0~36 数据信道上传输
```

广播包（ADV_IND 等）只在发现和建连阶段使用。31 字节 AdvData 装不下 ECDH 公钥（64 字节），也没有 L2CAP 信道做 SMP 交互。**加密协商是连接建立后才开始的。**

## Q8: CONNECT_IND 中的 1.25ms、SCA、Slave Latency、Supervision Timeout 分别是什么作用？

### 1.25ms — BLE 的时间基本单位

BLE 时基来自经典蓝牙的 625μs 时隙，两个时隙拼成最小步进：

```
1 时隙 = 625μs → 2 时隙 = 1.25ms ← 所有定时参数的"原子单位"
```

Interval、WinOffset、WinSize 都以 1.25ms 为步进。比如 Interval=24 → 实际 30ms（24 × 1.25）。没有这个统一粒度，双方无法用简单的整数约定时间。

### SCA（Sleep Clock Accuracy）— 睡眠时钟精度

设备在两次连接事件之间深度休眠，靠低精度低功耗的睡眠时钟计时唤醒。这个钟不准（误差可达 ±500 ppm）。

SCA 是 Master 告诉 Slave"我的钟有多不准"：

| SCA 值 | 精度 (ppm) | 意味 |
|--------|-----------|------|
| 0 | 251~500 | 差（廉价 RC 振荡器） |
| 7 | 0~20 | 好（外部晶振） |

Slave 根据 SCA 决定把接收窗口开多大才能兜住 Master 的时钟漂移。SCA 越差 → 窗口越大 → 功耗越高。

### Slave Latency — 从设备延迟次数

**允许 Slave 跳过 N 个连接事件不回复，Master 仍认为连接活着。** 没有数据时不用每个 Interval 都醒来发空包。

**硬约束**：`(1 + Latency) × Interval × 2 ≤ Timeout`

设 Latency=19、Interval=50ms → 每 1 秒才醒来一次 → 功耗降低约 20 倍。

### Supervision Timeout — 监督超时

**从"最后一次收到对方有效回复"开始倒计时，归零就宣布断开。** 范围 100ms ~ 32s（步进 10ms，不是 1.25ms）。短 → 快检测但易误断；长 → 稳定但断开感知慢。

### 四参数整体关系

```
Interval(30ms) → Interval → Interval →
┌────┐          ┌────┐     ┌────┐
│ TX │··········│ TX │·····│ TX │  (Slave N 次没回，第 N+1 次回了)
└────┘          └────┘     └────┘
◀─────────── Timeout(2s) ──────────▶ 期间一次没回 → 断开
```

核心方程：`(1+Latency) × Interval × 2 ≤ Timeout`——省电（Latency）和安全（Timeout）永远在博弈。

## Q9: 一个广播能同时发 ADV_IND 和 ADV_SCAN_IND 两种吗？还是互斥？

**互斥。每次广播事件只能发一种 PDU 类型。**

但关键点：**ADV_IND 本身就支持被扫描**，不需要切换成 ADV_SCAN_IND 也能回 SCAN_RSP：

```
一次广播事件（ADV_IND）：

  信道 37: 发 ADV_IND → 短暂开接收窗口
            有 SCAN_REQ → 回 SCAN_RSP（31 字节额外数据）
            有 CONNECT_IND → 进入连接态
            啥也没有 → 换信道 38
```

ADV_IND 同时支持"被扫描"和"被连接"——收到 SCAN_REQ 就回 SCAN_RSP，收到 CONNECT_IND 就进入连接。

**ADV_SCAN_IND 唯一的存在理由：想给额外广播数据但绝不允许连接。**

| | ADV_IND | ADV_SCAN_IND |
|---|---------|-------------|
| 能回 SCAN_RSP？ | ✅ | ✅ |
| 能被连接？ | ✅ | ❌ |
| 最大数据 | 31+31=62 字节 | 31+31=62 字节 |

不能"请求两次"——一次广播事件中每个信道只有一轮 SCAN_REQ→SCAN_RSP 的机会，设备回完就关窗换信道。

## Q10: AD Structure LTV 格式是什么？

广播数据（AdvData / ScanRspData）用 **LTV（Length-Type-Value）** 自描述格式编码，多个 AD Structure 依次拼接：

```
┌──────────┬──────────┬─────────────────────┐
│  Length  │ AD Type  │      AD Data        │
│  1 字节   │  1 字节   │  (Length-1 字节)    │
└──────────┴──────────┴─────────────────────┘
```

| 字段 | 大小 | 含义 |
|------|------|------|
| L（Length） | 1 字节 | Type + Data 的总长度（不含自身），= 0 表示结束 |
| T（Type） | 1 字节 | 数据类型，蓝牙 SIG 统一分配 |
| V（Value） | Length-1 字节 | 实际数据 |

### 实例拆解

```
02 01 06        ← Flags: 纯BLE + 普通可发现
03 03 0F 18     ← 16-bit Service UUID: 0x180F (Battery Service)
09 09 46 4C 4F 57 4D 45 54 45 52
                ← Complete Local Name: "FLOWMETER"
```

解析靠 Length 跳转：读 Length=2 → 跳 2 字节 → 读 Length=3 → 跳 3 字节 → 读 Length=9 → 跳 9 字节 → Length=0 结束。

### 常用 AD Type

| Type | 名称 | 说明 |
|------|------|------|
| **0x01** | Flags | LE 模式标志，**必须排第一个** |
| 0x03 | Complete 16-bit Service UUIDs | 每个 UUID 占 2 字节 |
| 0x09 | Complete Local Name | UTF-8 设备名 |
| 0x0A | TX Power Level | 1 字节有符号 dBm |
| 0xFF | Manufacturer Specific | 前 2 字节 Company ID + 自定义 |

## Q11: L2CAP Length 是纯长度吗？最大能传 65535 字节？

**L2CAP Length 是纯长度，不含任何标志位。**

```
┌──────────────────┬──────────────────┬─────────────────────────┐
│    Length        │       CID        │    Information Payload  │
│    (2 字节)       │     (2 字节)      │    (Length 字节)        │
│  纯数字，无标志    │   协议类型         │    上层 PDU              │
└──────────────────┴──────────────────┴─────────────────────────┘
```

Length 是 2 字节无符号整数 → 纸面上限 **65535 字节**。但实际瓶颈不在 Length 字段：

| 层 | 限制 | 典型值 |
|----|------|--------|
| L2CAP Length | 65535 | 纸面上限 |
| ATT MTU | 协商决定 | Android 517, iOS 527 |
| LL Payload | 链路层协商 | 27（4.0）→ 251（4.2+） |

**实际一次 `writeCharacteristic()` 能发多少由 ATT MTU 决定，不是 L2CAP 的 65535。** 类型 → CID 管，分片 → 链路层 LLID 管，加密 → 链路层 MIC 管。Length 只管一件事：后面跟了多少字节。

## Q12: 数据信道 Header 中的 SN 和 NESN 只有 1 bit，为什么能保证不丢包？

**因为 BLE 链路层是停等协议（Stop-and-Wait），不是 TCP 那种滑动窗口。** 一次只有一个包在路上，1 bit SN 只用来区分"新包"和"重传"两种状态。

### BLE 停等 vs TCP 滑动窗口

```
TCP 滑动窗口：
  多包同时在路上 → 可能丢中间某个 → 需要大 SN 分辨丢了哪个
  [SN=1001][1002][1003] ──→ 丢 1002 → ACK=1002 → 从 1002 起重传

BLE 停等协议：
  发一包 → 等确认 → 发下一包
  最多只有一个包在飞，只需要区分"这个包"和"下一个包"
```

### SN：翻转 = 新数据，不变 = 重传

```
正常:
  事件1: Master → Slave: SN=0 [新数据]
         Slave → Master: NESN=1 [收到了]
  事件2: Master → Slave: SN=1 [新数据，翻了一下]
         Slave → Master: NESN=0 [收到了]
  事件3: Master → Slave: SN=0 [又翻了，新数据]

丢包:
  事件1: Master → Slave: SN=0 [新数据]
         Slave → Master: NESN=1 [收到了！]
         ↑ 这个回复在空中丢了！Master 没收到
         
         Master 超时（150μs 内没回）→ 重传
  事件1':Master → Slave: SN=0 [SN 没变！]
         Slave 收到: SN=0 → 和上次一样 → 重传，不处理数据，只重发 ACK
```

### NESN：隐含 ACK

NESN 翻转（0→1 或 1→0）= ACK："下一个用这个 SN"= 上一个收到了。NESN 不变 = NAK："还是那个 SN"= 没收到。

| | TCP | BLE 链路层 |
|---|-----|----------|
| 协议类型 | 滑动窗口 | 停等 |
| 同时在路上的包 | 多个 | 最多 1 个 |
| SN 大小 | 32 bit | 1 bit |
| ACK 方式 | 独立 ACK 包 | NESN 翻转隐含 ACK |
| 乱序 | 可能 | 不存在 |

## Q13: LL Control PDU 的类型怎么区分？和 L2CAP Signaling 有什么区别？

### LL Control PDU 通过 Opcode（Payload 第一个字节）区分

```
LLID = 11 → 链路层直接处理，不交给 L2CAP

Control Payload 格式:
  [Opcode(1B)] [Command Data(变长)]
  
收到 Payload[0]:
  0x00 → LL_CONNECTION_UPDATE_IND  → 11 字节数据
  0x01 → LL_CHANNEL_MAP_REQ        → 7 字节数据
  0x02 → LL_TERMINATE_IND          → 1 字节 ErrorCode
  0x03 → LL_ENC_REQ                → 22 字节数据
  0x0D → LL_REJECT_IND             → 2 字节数据
```

### 和 L2CAP Signaling (CID=0x0005) 是两条不同的路

```
LL Control PDU:
  LLID=11 → Payload 以 Opcode 开头 → 不出链路层 → Controller 内部消化 → 没有 CID

L2CAP Signaling:
  LLID=10 → L2CAP → CID=0x0005 → Host 协议栈处理 → 有 Identifier 做请求匹配
```

| | LL Control | L2CAP Signaling |
|---|---|---|
| 处理层级 | Controller（链路层） | Host（L2CAP 层） |
| 怎么识别 | LLID = 11 | LLID = 10 → CID = 0x0005 |
| 有 CID 吗？ | ❌ 不过 L2CAP 头 | ✅ CID=0x0005 |
| 连接参数更新 | Master 发起 | **Slave 也可以发起** |
| 有事务 ID？ | 无 | 有 Identifier 字段 |

Slave 可以走 L2CAP Signaling 发起参数更新请求——这是它唯一能"主动说话"的通道。

## Q14: BLE 加密是必须的吗？可以不经过 SMP 直接发 ATT/GATT 数据？

**不必须。连接建立后链路就是明文的，ATT/GATT 数据可以直接裸奔。**

```
连接建立
  ═══════ Connection State ═══════
  │
  ├── 链路明文，ATT/GATT 直接可用
  │
  └── 只有在访问"需加密"的属性时才会触发 SMP 配对
```

### 按需触发加密

```
readCharacteristic(Handle=0x0010) → 公开属性 → ✅ 直接返回
readCharacteristic(Handle=0x0030) → 设了"需加密"
  → ATT Error: Insufficient Encryption
  → Android 协议栈自动触发配对
  → SMP → LL_ENC_REQ/RSP → 链路加密
  → 自动重试读 0x0030 → ✅ 成功
```

Permission 存在每个 Attribute 上，不是连接级别一刀切。同一个连接里公开数据和加密数据可以共存。

## Q15: AES 是对称加密吗？

**是的——加密和解密用同一把钥匙。**

```
非对称 (RSA/ECC):              对称 (AES):
  公钥🔓 加密 → 私钥🔐 解密       同一把钥匙🔑 加密 + 解密
  两把不同的钥匙                  双方必须都知道这把钥匙
```

### AES 在 BLE 中的位置

BLE 的加密方案是 **AES-CCM**：
- **AES**：对称加密算法（128-bit 密钥）
- **CCM**：工作模式 = CTR（加密）+ CBC-MAC（生成 MIC，防篡改）

**BLE 的策略是混合的**：配对阶段用 ECDH（非对称）安全协商出对称密钥 → 数据阶段用 AES（对称）高效加密。因为对称加密比非对称快得多、省电得多。

## Q16: SMP 配对特征交换中的 Init Key / Resp Key 是什么？

**"我手里有哪些钥匙，等加密隧道建立后分给你"的预告清单。** 每个 bit 对应一种密钥类型：

| bit | 密钥 | 含义 |
|-----|------|------|
| 0 | **LTK** | 加密链路的长期密钥 |
| 1 | **IRK** | 身份解析密钥（认出 RPA 地址） |
| 2 | **CSRK** | 数据签名密钥 |
| 3~7 | RFU | 保留 |

### 典型组合

```
双方都填 0x03 (LTK + IRK):
  → Phase 4 互相分发 LTK + IRK → 绑定完成 → 下次自动重连

Peripheral 填 0x00（什么都不给）:
  → Phase 4 跳过 → 本次加密完各走各的，不绑定

Phone 给 IRK, Peripheral 不给 IRK:
  → Phone 需要隐私（地址每 15 分钟换），Peripheral（固定地址）不需要
```

Init Key = 发起方将要分发的，Resp Key = 发起方希望响应方分发的。Phase 1 的承诺，Phase 4 的履行。

## Q17: AuthReq 中的 Bonding、MITM、SC 分别是什么意思？

AuthReq（1 字节）的三个标志决定了"怎么配对、配成什么样"：

```
AuthReq bit:
  [0] Bonding → 配完要不要存入 Flash？(0=不存, 1=绑定)
  [1] MITM    → 要不要防中间人？(0=Just Works, 1=需用户验证)
  [2] SC      → 用什么密码学？(0=Legacy AES-CMAC, 1=ECDH P-256)
```

| 标志 | = 0 | = 1 |
|------|-----|------|
| **Bonding** | 本次加密完就忘，下次重连重新配对 | 密钥存 Flash，下次自动加密 |
| **MITM** | 不验证对方身份（Just Works） | 用户参与验证（Numeric Comparison / Passkey） |
| **SC** | Legacy（AES-CMAC, ~20-bit 强度） | Secure Connections（ECDH, 128-bit 强度） |

### 组合效果

| Bonding | MITM | SC | 结果 |
|---------|------|-----|------|
| 0 | 0 | 0 | Legacy Just Works，不存密钥 |
| 1 | 0 | 1 | SC Just Works——加密了但不知道跟谁 |
| 1 | 1 | 1 | SC Numeric Comparison——全功能最强安全 |
| 1 | 1 | 0 | Legacy Passkey——6 位 PIN 验证 |

## Q18: Legacy Pairing（不支持 SC）如何靠 6 位 PIN 生成 AES 密钥？

**核心思路：用一个弱秘密（TK）临时保护强密钥（LTK）的分发，然后扔掉弱秘密。**

### 6 位 PIN 是什么

用户看到/输入的 6 位十进制数字（000000~999999），安全强度仅 ~20 bit。双方各自把 PIN 转成 128-bit TK，从不在空中发送。

### 完整流程（六步）

**第一步 — Phase 1 特征交换**：明文交换 IO Capabilities、AuthReq 等，协商配对方式（Just Works / Passkey / OOB）。

**第二步 — 获取 TK**：
- Just Works: TK = 0x0000...0000（全零，攻击者也知道）
- Passkey: TK = PIN 扩展为 128-bit
- OOB: TK = NFC 传输的 128-bit 随机数

**第三步 — Confirm-Random（承诺-兑现）确认双方 TK 一致**：

```
3a. 先发承诺（锁盒子）:
  设备 A: Mconfirm = AES-CMAC(Key=TK, Data=Mrand || 配对参数)
  设备 B: Sconfirm = AES-CMAC(Key=TK, Data=Srand || 配对参数)
  
  此时 Mrand/Srand 还是保密的——攻击者拿到 Mconfirm 但无法反推 Mrand

3b. 再揭晓随机数（开盒子）:
  A 发 Mrand(明文), B 发 Srand(明文)
  
  收到后各自验证:
  AES-CMAC(Key=TK, Data=Mrand || 参数) =?= 之前收的 Mconfirm
  AES-CMAC(Key=TK, Data=Srand || 参数) =?= 之前收的 Sconfirm

  一致 → "对方确实知道 TK"
  
  ⚠️ 只证明对方知道 TK，不证明对方是谁
     如果 TK=0 → 任何人都能算对 → 这个验证毫无意义
```

**第四步 — 生成 STK**：

```
STK = AES-CMAC(Key=TK, Data=Mrand || Srand)
→ 256-bit 随机数"搅拌"进 TK → 输出均匀的 128-bit 值
→ 随机性来自 Mrand/Srand，不来自弱 TK
```

**第五步 — Phase 3：STK 加密分发 LTK**：

```
═══ STK 加密 ═══
A → B: LTK(128-bit 随机), EDIV+Rand(索引)
═══ STK 扔掉，永不再用 ═══

下次连接 → LL_ENC_REQ(EDIV+Rand) → 直接拿 LTK → 加密
```

**第六步 — 下次重连**：用户无感知，跳过全部配对流程。

### 安全弱点

| 弱点 | 原因 |
|------|------|
| Just Works 完全裸奔 | TK=0 人人知道 → 攻击者抓到 Mrand/Srand 直接算 STK → 全解密 |
| Passkey 仅 20-bit | 6 位 PIN = 10⁶ 种可能 → 暴力破解几秒完成 |
| 无前向安全性 | LTK 静态不变，一旦泄露历史全部解密 |
| 随机数明文发出 | Mrand/Srand 空中可截获 |

→ 蓝牙 4.2 引入 LE Secure Connections（ECDH P-256）解决这些问题。

## Q19: ATT/GATT 每个属性都有一个 Handle，是通过 Handle 去取数据吗？每个 Handle 指向一段不定长的数据区域？

**是的，全部 ATT 读写操作都通过 Handle 寻址。Handle 是 16 位查找键（类似数组下标），不是 C 指针。每个 Handle 对应的 Value 长度可变，取决于该属性的 Type 和实际内容。**

### Handle 就是属性表的"行号"

GATT Server 内部维护一张属性表（Attribute Table），Handle 是这张表的索引：

```
Handle  │ Type (UUID)       │ Permissions        │ Value
────────┼───────────────────┼────────────────────┼──────────────────
0x0001  │ 0x2800 (Primary)  │ Read Only          │ 0x180A (Device Info Service UUID)
0x0002  │ 0x2803 (CharDecl) │ Read Only          │ [Props=0x02, ValueHandle=0x0003, UUID=0x2A29]
0x0003  │ 0x2A29 (Manuf.)   │ Read Only          │ "OPPO" (变长字符串，最多 512 字节)
0x0004  │ 0x2902 (CCCD)     │ Read + Write       │ 0x0000 (固定 2 字节)
...     │ ...               │ ...                │ ...
```

Handle 是 16 位无符号整数（0x0001~0xFFFF，0x0000 保留）。Client 说"读 Handle 0x0003"，Server 查 `table[0x0003]` → 校验权限 → 返回值。

### 每次 Handle 分配多少空间？

**没有固定大小，每个属性独立决定：**

| 属性类型 | 典型 Value 大小 | 说明 |
|---------|---------------|------|
| Primary/Secondary Service (0x2800/0x2801) | 2 或 16 字节 | 内含 Service UUID |
| Characteristic Declaration (0x2803) | 5 字节固定 | Properties(1B) + ValueHandle(2B) + UUID(2B) |
| Characteristic Value | 可变（0~512 字节） | 实际数据，如设备名、电池电量 |
| CCCD (0x2902) | 2 字节固定 | Notification/Indication 开关 |
| Report Reference (0x2908) | 2 字节固定 | Report ID + Type |

### 单次能读多少？Handle 单次寻址范围

**单次 ATT Read/Write 能传输的字节数：**

```
单次可传输 = ATT MTU - 3
              ↑         ↑
          协商决定    Opcode(1B) + Handle(2B) 的开销
```

例如 MTU=251 → 单次最多读 248 字节。如果 Value 更大（如固件 OTA 数据），需用 Read Blob Request（带 Offset 参数，分段续读）。

### Handle 空间的全局视野

```
Client 视角的服务发现过程：
  "Read By Group Type" → 拿到 Service 列表
    Handle 0x0001~0x0005 → Device Info Service (0x180A)
    Handle 0x0006~0x0010 → Battery Service (0x180F)
  "Read By Type" → 在每个 Service 范围内发现 Characteristic
    Handle 0x0002 → Manufacturer Name String (Properties=Read, ValueHandle=0x0003)
    Handle 0x0007 → Battery Level (Properties=Read+Notify, ValueHandle=0x0008)
  "Find Information" → 发现 Descriptor
    Handle 0x0004 → CCCD (0x2902, 控制 0x0003 的通知)
    Handle 0x0009 → CCCD (0x2902, 控制 0x0008 的通知)
  "Write Request" → Handle=0x0009, Value=0x0001 → 开启 Battery Level 通知
```

**类比**：Handle 像数组下标，Attribute Table 是 GATT Server 的"大数组"。`read(handle)` = `table[handle].value`。GATT Server 不暴露内存地址，Client 只能通过 Handle 间接访问——这是协议层的安全边界。
