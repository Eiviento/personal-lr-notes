# 6.5 服务发现流程

## 一、为什么需要

Client 连接后不知道 Server 的 GATT 表结构 → 用 ATT Read Request 一步步"探"出来。

## 二、三个 Round

**Round 1**：READ_BY_GROUP_TYPE (0x2800) → 发现所有 Primary Service（UUID + Handle 起始/结束范围）

**Round 2**：READ_BY_TYPE (0x2803)，每个 Service 内部 → 发现所有 Characteristic（Declaration Handle + Properties + Value Handle + UUID）

**Round 3**：FIND_INFO，扫描 Handle 间隙 → 发现所有 Descriptor（CCCD/CUD等）

## 三、完成后的缓存

`getServices()` → `List<BluetoothGattService>` → `getCharacteristics()` → `getDescriptors()`

读写操作直接走缓存，无需重新发现（除非 `refresh()`）。

## 四、为什么 discoverServices() 慢？

- 每个 Service 3~4 轮往返
- Connection Interval 慢时（100ms），20 个包 = 2 秒
- 加密未启动 → ATT 被拒 → 协议栈先启动加密再重试

## 五、优化建议

- 设备固件 Service 别超过 5 个，Characteristic 控制在 15 个以内
- 连接成功后立刻调用 discoverServices()
- `onServicesDiscovered()` 回调前不做 GATT 操作
- 不硬编码 Handle → 靠服务发现
