# 3.1 链路层状态机

## 一、为什么重要

链路层本质就是一个状态机。BLE 设备在任何时刻一定处于 5 个状态之一。Android API 里的扫描、连接、断开，底层全是状态迁移的反映。

## 二、五个状态

```
                ┌─────────────┐
                │   STANDBY   │
                │   空闲状态    │
                └──┬──┬───┬──┘
       ┌───────────┘  │   └───────────┐
       ▼              ▼               ▼
┌─────────────┐ ┌──────────┐  ┌─────────────┐
│ ADVERTISING │ │ SCANNING │  │ INITIATING  │
│   广播状态   │ │  扫描状态 │  │  发起连接    │
└──────┬──────┘ └──────────┘  └──────┬──────┘
       │                             │
       └──────────┬──────────────────┘
                  ▼
          ┌──────────────┐
          │  CONNECTION  │
          │   连接状态     │
          └──────────────┘
```

关键约束：同一时刻只能处于一个状态；跨状态必须经过 Standby。

## 三、五个状态详解

**Standby（空闲）**：不收不发，功耗最低。上电默认进入，从任何状态断开后回到这里。

**Advertising（广播）**：周期性发广播包，"我在这里"。只发不收（除了 CONNECT_IND）。被连接后进入 Connection（做 Slave）。

**Scanning（扫描）**：监听广播信道，收集广播包。只收不发（除了主动扫描的 SCAN_REQ）。不能直接发起连接。

**Initiating（发起连接）**：定向等待特定设备的广播，收到后发 CONNECT_IND。和 Scanning 的区别：Scanning 广泛监听，Initiating 定向等待。成功后进入 Connection（做 Master）。

**Connection（连接）**：按 Connection Interval 周期性交换数据。Master 控制节奏，Slave 响应。退出条件：LL_TERMINATE_IND / Supervision Timeout / 上层命令。

## 四、状态迁移汇总

| 从 | 到 | 触发条件 |
|----|-----|---------|
| Standby | Advertising | 开始广播 |
| Standby | Scanning | 开始扫描 |
| Standby | Initiating | 连接 XX 设备 |
| Advertising | Standby | 停止广播 |
| Advertising | Connection | 收到 CONNECT_IND（做 Slave） |
| Scanning | Standby | 停止扫描 |
| Initiating | Standby | 超时 |
| Initiating | Connection | 收到目标广播后发 CONNECT_IND（做 Master） |
| Connection | Standby | 断连 / 超时 / 命令 |

## 五、对 Android 开发的影响

| Android API | 底层状态迁移 |
|------------|-------------|
| `startScan()` | Standby → Scanning |
| `stopScan()` | Scanning → Standby |
| `connectGatt()` | Standby → Initiating → Connection |
| `disconnect()` | Connection → Standby |

重要启发：不能同时扫描和发起连接——必须先 stopScan 再 connectGatt。
