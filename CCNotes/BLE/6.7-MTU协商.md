# 6.7 MTU 协商

## 一、MTU 定义

ATT MTU = ATT 层单个包最大 Payload。默认 23 字节，可协商至 517。

不同于 LL Payload (27/251) 和 L2CAP PDU（取决于 LL）。

## 二、默认 MTU=23 的瓶颈

ATT_READ_RSP 可用 22B，ATT_WRITE_REQ 可用 20B，ATT_NOT 可用 20B。**每次通知只能传 20 字节。**

## 三、MTU 协商

```
ATT_EXCHANGE_MTU_REQ (ClientRxMTU=512)
ATT_EXCHANGE_MTU_RSP (ServerRxMTU=251)
→ ATT MTU = min(512, 251) = 251
```

Android: `gatt.requestMtu(512)` → 回调 `onMtuChanged(mtu, status)`

## 四、长属性读写

**Read Blob**：Read Request 拿前 250B → Read Blob Req 续读 → ... → Response 长度 < MTU-1 表示结束

**Prepare+Execute Write**：多发 Prepare Write Req（不生效）→ Execute Write Req（原子提交）或取消失败。Android `writeCharacteristic()` 不支持自动分片，超 MTU-3 返回 false。

## 五、MTU 与吞吐

MTU=251 比 23 提升 2~4 倍（减少 ATT PDU IFS 等待和 L2CAP 头开销）。

## 六、最佳实践

```java
onConnectionStateChange(CONNECTED) → requestMtu(512)
onMtuChanged(SUCCESS) → discoverServices()
```

**必须先 requestMtu 再 discoverServices**，后续所有操作享受大 MTU。

## 七、关键注意

- 部分设备不支持大 MTU
- 长写 Android API 不自带分片，超 MTU-3 必须自己切片
- 长读 Android API 自动处理 Read Blob（App 无感）
