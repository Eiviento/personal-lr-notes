# 3.5 连接建立全过程

## 一、完整时序

```
Peripheral (Advertising)              Central (Scanning→Initiating)
  ADV_IND (Ch37/38/39)  ──────────→  扫描到
                                      stopScan() → Initiating
  ADV_IND ────────────────────────→  收到！
  ←── CONNECT_IND (同一信道) ──────  发出连接请求
  进入 Connection (Slave)            进入 Connection (Master)
  ==== 第一次连接事件 ====
```

CONNECT_IND 必须在同一广播事件内发出（设备广播后监听窗口仅百微秒级）。从扫描到进入连接可短至 < 10ms。

## 二、CONNECT_IND 的 36 字节参数

| 参数 | 大小 | 含义 |
|------|------|------|
| Access Address | 4 B | 连接"会话 ID"，随机生成，链路层靠它区分不同连接 |
| CRC Init | 3 B | CRC 校验初始值 |
| WinSize | 1 B | 传输窗口大小 |
| WinOffset | 2 B | 距第一次连接事件的延迟 |
| **Interval** | 2 B | Connection Interval |
| **Latency** | 2 B | Slave Latency |
| **Timeout** | 2 B | Supervision Timeout |
| Channel Map | 5 B | 37 数据信道可用位图 |
| Hop Increment | 1 B | 跳频算法种子 |
| SCA | 1 B | 时钟精度补偿 |

## 三、核心参数详解

### Connection Interval
范围：7.5ms ~ 4000ms（6~3200 × 1.25ms），典型 30~100ms。

| Interval | 适用场景 |
|----------|---------|
| 7.5~15ms | OTA 升级 |
| 30~50ms | 流量计日常读数（推荐） |
| 100~500ms | 周期性慢速采集 |
| 1000ms+ | 小时级长期监测 |

### Slave Latency
Slave 可跳过不回复的连接事件数。约束：`(1+Latency)×Interval×2 ≤ Timeout`。

没有 Latency 时每个 Interval 都要醒来，即使没数据可发。设 Latency=19、Interval=50ms 时，1 秒无数据则跳过 19 个空周期，功耗降低 ~20 倍。**但有数据可随时醒来发。**

### Supervision Timeout
连续无响应超时后宣布断开。范围 100ms ~ 32s。短 → 快检测但易误断；长 → 稳定但断开感知慢。

### Channel Map
37-bit 位图，1=可用，0=跳过。初始全 1（全可用），后续根据干扰动态更新（见 3.6）。

### Hop Increment
`next_channel = (current + hopIncrement) % 37`，取值 5~16，确保遍历全部 37 个信道。

## 四、Master/Slave 职责

- **Master**：控制 Interval 节奏（每次事件先发包）、管理 Channel Map、发起参数更新
- **Slave**：收到 Master 包 150μs 内回复、可通过 Latency 跳过空周期省电

## 五、对 Android 开发的影响

| 参数 | Android API |
|------|------------|
| Conn Interval | `requestConnectionPriority(HIGH/BALANCED/LOW_POWER)`，最终决定权在设备 |
| Slave Latency | App 无法直接控制 |
| Timeout | 影响 `onConnectionStateChange(DISCONNECTED)` 的延迟 |
| Channel Map/Hop | 完全透明，不可见不可控 |

常见踩坑：`CONNECTION_PRIORITY_HIGH` 请求可能被设备忽略，需用 `onConnectionUpdated()` 回调确认。
