# 6.1 ATT 协议基础

## 一、ATT 位置

ATT 是 BLE 数据交互的"语言"，定义 Client 如何向 Server 读写数据的动词和语法。承载于 L2CAP CID=0x0004。

## 二、Client-Server 模型

- Server（通常是 Peripheral/流量计）：拥有 Attribute Database
- Client（通常是 Central/手机）：发送 ATT 请求读写数据
- 角色可互换（规范支持，但极少用）

## 三、Attribute 结构

```
[Handle(2B)] [Type(UUID)] [Value(变长0~512B)] [Permissions(独立存储)]
```

**Handle**：唯一编号（0x0001~0xFFFF），相当于主键。固件预定义，每次连接不变但不同固件版本可能不同（所以需要服务发现）。

**Type**：UUID 标识这是什么数据（16-bit 标准或 128-bit 自定义）。

**Permissions**：独立于 Value，控制访问条件。每种操作（Read/Write）可独立设置：无要求 / 需加密 / 需认证 / 需授权。

## 四、ATT PDU 通用格式

```
[Opcode(1B)] [Parameters(可变)]
Opcode: Bit 0~5=Method, Bit 6=Command Flag(0=需响应/1=无需响应), Bit 7=Signature Flag
```

## 五、数据旅程

```
readCharacteristic(handle=0x0017)
→ ATT_READ_REQ[Opcode=0x0A][Handle=0x0017] (3B)
→ L2CAP[CID=0x0004] (+4B=7B)
→ HCI ACL (+4B=11B)
→ LL Data (+4B=15B)
→ PHY → 无线电
```

## 六、对开发的影响

- Handle ↔ `BluetoothGattCharacteristic`
- readCharacteristic 返回 false → 可能权限不够，需先配对加密
- 不要硬编码 Handle → 用 UUID 做 GATT 服务发现
- Client/Server 可互换但极少使用
