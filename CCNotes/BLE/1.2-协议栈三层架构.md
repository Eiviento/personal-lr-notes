# 1.2 协议栈三层架构：Controller / Host / Application

## 一、为什么需要分层？

BLE 协议不是一块铁板，而是像一栋楼一样分层的。分层的好处：

1. **每层独立演进**：物理层换了调制方式，不影响上层 GATT 读写数据
2. **厂商可以各做各的**：芯片厂（TI、Nordic、Dialog）做底层，Android/iOS 做上层
3. **你作为应用开发者，只需要关注最上面几层**

## 三、每层的职责（一张表说清楚）

| 层 | 位置 | 职责 | 打比方 |
|---|------|------|--------|
| **PHY** | Controller | 把 0/1 变成无线电波发出去，反之亦然 | 邮局的卡车，负责物理运输 |
| **Link Layer** | Controller | 管理广播/扫描/连接状态，跳频，校验，加密 | 邮局调度中心，决定什么时候发、走哪条路 |
| **HCI** | 中间桥梁 | Controller 和 Host 之间的通信协议（通常走 UART/USB） | 调度中心和邮局管理系统的电话线 |
| **L2CAP** | Host | 把上层数据打包/拆包，多路复用（区分 SMP/ATT 等不同上层协议） | 分拣中心，区分包裹类型并分包 |
| **SM** | Host | 配对认证、密钥生成、加密链路 | 安检+密码科 |
| **ATT** | Host | 定义数据的读写规则（Client/Server、Handle、权限） | 数据操作语言（SELECT/UPDATE 之于数据库） |
| **GATT** | Host | 在 ATT 之上定义数据的组织形式（Service/Characteristic 树） | 数据库的表结构定义（Schema） |
| **GAP** | Host | 定义设备角色（Central/Peripheral 等）、广播、发现、连接策略 | 设备社交规范 |
| **Application** | App | 你的 Android App 调用 BLE API 实现业务逻辑 | 最终用户 |

## 四、HCI：Controller 和 Host 的分界线

这是理解 BLE 架构最关键的概念之一。

### 4.1 为什么有 HCI？

在实际硬件中，Controller 和 Host 可能在同一块芯片上，也可能分开：

**双芯片方案**（早期/高端设备）：
```
┌──────────┐    UART/USB    ┌──────────────┐
│  Host    │←─────HCI─────→│  Controller  │
│ (主控 CPU)│               │ (蓝牙芯片)    │
│ Android   │               │  如 CSR/BCM  │
│ 跑在主CPU  │               │  独立的蓝牙IC │
└──────────┘               └──────────────┘
```

**单芯片方案**（现在主流 IoT）：
```
┌─────────────────────────┐
│  Host + Controller      │
│  在同一块 SoC 上         │  ← 如 Nordic nRF52 系列、TI CC254x、ESP32
│  HCI 变成内部函数调用    │
└─────────────────────────┘
```

> **你的 Android 手机上是双芯片方案**：Android 系统（Host）跑在主 CPU 上，蓝牙芯片（Controller）是独立的硬件，之间通过 UART 上的 HCI 协议通信。

### 4.2 HCI 的三类包

| 包类型 | 方向 | 举例 |
|--------|------|------|
| **HCI Command** | Host → Controller | "开始扫描"、"建立连接" |
| **HCI Event** | Controller → Host | "扫描到了设备"、"连接建立成功" |
| **HCI ACL Data** | 双向 | 承载 ATT/GATT 的实际数据 |

## 五、从 Android 开发者的视角看

当你调用 Android BLE API 时，调用链是这样的：

```
你的代码：
  BluetoothGatt.readCharacteristic(characteristic)
       │
       ▼
Android BLE Framework (BluetoothGatt.java)
       │
       ▼
Android Bluetooth Stack (Bluedroid / Fluoride)
       │  — 这里跑的是 GATT/ATT/SM/L2CAP（Host 层）
       │
       ▼
HCI (UART 发 HCI Command)
       │
       ▼
蓝牙芯片固件 (Link Layer + PHY)  — Controller
       │
       ▼
无线电波发出
```

**所以你的 Android API 调用，最终变成了一串 HCI Command 发给蓝牙芯片，芯片再通过无线电发出去。**

## 六、关于 BLE 协议栈的开源实现

Android 系统内置的 BLE 协议栈是 **Bluedroid**（Android 4.2+）或 **Fluoride**（Android 后续版本的名字），它是 Google 维护的开源实现。

除了手机上的协议栈，还有一些适合嵌入式/学习的开源 BLE 协议栈：

| 协议栈 | 特点 |
|--------|------|
| **Zephyr BLE** | Linux 基金会的开源 RTOS 的 BLE 栈，代码质量高，适合学习 |
| **NimBLE** | Apache 开源，轻量级 Host + Controller，可运行在 Linux/嵌入式 |
| **BlueZ** | Linux 官方的蓝牙协议栈（包含经典蓝牙 + BLE） |
| **btstack** | 另一套开源协议栈 |

> 未来深入学习时，可以对照这些开源代码理解协议细节。

---

## 七、一句话总结

> **Controller（PHY + Link Layer）管"怎么发"，Host（L2CAP + SM + ATT/GATT/GAP）管"发什么"和"跟谁发"，HCI 是它们之间的翻译官。你的 Android App 站在最顶层，通过 API 逐层调用到底层无线电。**
