# 1.3 核心概念速览

> 目的：在深入每一层之前，先把 BLE 的核心概念快速串一遍，建立一个"各概念之间是什么关系"的全局地图。

---

## 概念地图

```
                    一个 BLE 交互的完整生命周期

  广播阶段                连接阶段               数据交互阶段
  ─────────              ─────────              ─────────────

  Peripheral             Central                Central ↔ Peripheral
  (设备/流量计)           (手机)                  (已连接状态)

    │                      │                       │
    │  ① 广播 (Advertising)│                       │
    │  "我在这里！"         │                       │
    │════════════════════→│                       │
    │                      │                       │
    │                      │  ② 扫描 (Scanning)     │
    │                      │  "让我看看周围有谁"     │
    │                      │                       │
    │                      │  ③ 发起连接            │
    │                      │══════════════════════→│
    │                      │                       │
    │                      │         ④ 服务发现     │
    │                      │   "你能提供什么数据？"  │
    │                      │══════════════════════→│
    │                      │←══════════════════════│
    │                      │   "我有温度、电量..."  │
    │                      │                       │
    │                      │         ⑤ 数据交互     │
    │                      │   读/写/通知          │
    │                      │══════════════════════→│
    │                      │←══════════════════════│
    │                      │                       │
```

---

## 一、广播（Advertising）— "我在这里！"

设备（Peripheral）周期性向外喊话，内容极少（最多 31 字节）：

```
"我是流量计，型号 ABC，你可以连我"
```

**关键点**：
- 广播由 **Peripheral**（设备端）发出
- 在 3 个专用广播信道上轮流转发
- 间隔通常为 20ms ~ 几秒
- 数据极短（31 字节，5.0 后可以扩展到更多）
- 功耗极低：喊一句就睡，睡醒了再喊

---

## 二、扫描（Scanning）— "让我看看周围有谁"

手机（Central）监听广播信道，收集附近的广播包。

**关键点**：
- 扫描由 **Central**（通常是手机）执行
- 被动扫描：只听不回应（省电）
- 主动扫描：听到广播后发 SCAN_REQ，设备回 SCAN_RSP（多拿 31 字节数据）
- 扫描结果包含：设备地址、RSSI（信号强度）、广播数据内容

---

## 三、连接（Connection）— "我们建立通信"

手机发现目标设备后，发送 CONNECT_IND 请求，双方进入连接状态。

**关键点**：
- 连接是**点对点**的，一个 Peripheral 同一时间通常只连一个 Central
- 连接后，双方约定一个**连接间隔**（Connection Interval，如 30ms）
- 每个间隔内，Master 先发一个包，Slave 必须回复
- 连接后**不再用广播信道**，改用 37 个数据信道 + 跳频

---

## 四、GATT 数据组织 — "数据怎么存"

这是 BLE 数据最核心的组织方式。想象成一个小型数据库：

```
Profile（不是具体概念，是"全套服务的总称"）
│
├── Service (服务)              ← 一个大功能模块
│   │   UUID: 0x180F            ← 标准电池服务
│   │
│   ├── Characteristic (特征)   ← 一个具体数据项
│   │   │   UUID: 0x2A19        ← 电池电量
│   │   │   Properties: Read | Notify  ← 可读 + 可通知
│   │   │   Value: 85           ← 85%
│   │   │
│   │   └── Descriptor (描述符) ← 附加配置
│   │       │   CCCD            ← Client Characteristic Configuration
│   │       │                    ← 写 0x0001 = 开启通知
│   │       │
│   ├── Characteristic
│   │   │   UUID: ...           ← 其他特征
│   │   │   ...
│   │
├── Service
│   │   UUID: 自定义 128-bit    ← 厂商自定义服务（如流量计服务）
│   │   ├── Characteristic: 流速
│   │   ├── Characteristic: 累计流量
│   │   ├── Characteristic: 校准参数
│   │   └── ...
```

### 概念速记

| 概念 | 一句话 | 类比 |
|------|--------|------|
| **Profile** | 一系列 Service 的集合，描述一个完整的应用场景 | 数据库 |
| **Service** | 一个功能模块 | 数据库的一张表（如"电池信息表"） |
| **Characteristic** | 一个具体的数据项 | 表中的一列（如"电量百分比"） |
| **Descriptor** | Characteristic 的附属配置信息 | 列的元数据（如"这个字段的单位是%"） |
| **CCCD** | 最常用的 Descriptor，控制是否开启通知/指示 | "有新数据时主动通知我"的开关 |
| **UUID** | 标识 Service/Characteristic 的唯一 ID | 表名和列名（16-bit = 标准名，128-bit = 自定义名） |

## 五、ATT 数据交互 — "数据怎么读写"

ATT (Attribute Protocol) 定义了 Client 如何操作 Server 上的 Attribute。

| 操作 | 方向 | 说明 | 有响应？ |
|------|------|------|---------|
| **Read** | Client → Server | 读取一个 Characteristic 的值 | ✓ |
| **Write Request** | Client → Server | 写入值，等待确认 | ✓ |
| **Write Command** | Client → Server | 写入值，不等待确认 | ✗ |
| **Notification** | Server → Client | Server 主动推送数据 | ✗ |
| **Indication** | Server → Client | Server 推送数据，等待 Client 确认 | ✓ |

**读/写：手机主动问/改**
```
手机: "你的电量是多少？"  (Read Request)
设备: "85%"               (Read Response)

手机: "把采样间隔改成 5 秒" (Write Request)
设备: "好的，改好了"       (Write Response)
```

**通知/指示：设备主动报**
```
设备: 流量变化了！→ 自动发给手机 (Notification)
设备: 报警了！→ 发给手机并要求确认 (Indication)
手机: "收到"                        (Confirmation)
```

## 六、GAP 角色 — "谁是谁"

GAP (Generic Access Profile) 定义了 BLE 设备的四种角色：

| 角色 | 干什么 | 通常是 |
|------|--------|--------|
| **Broadcaster** | 只广播，不可连接 | 信标（Beacon）、温湿度传感器（只发不收） |
| **Observer** | 只扫描监听，不发起连接 | 数据采集网关 |
| **Peripheral** | 广播 + 可被连接（Slave） | **你们的流量计** |
| **Central** | 扫描 + 发起连接（Master） | **你的 Android 手机** |

**典型组合**：Peripheral（流量计）←→ Central（手机）

## 八、你现在只需要记住

> **BLE 的交互就是：设备广播 → 手机扫描发现 → 建立连接 → 手机发现设备的 GATT 表（有什么 Service、什么 Characteristic）→ 读写数据 / 订阅通知 → 断开。**

后面每一层深入学习时，时刻回想这张概念地图，知道自己在学的是整个流程的哪一环。
