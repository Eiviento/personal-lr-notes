# 5.4 LE Secure Connections 详解

## 一、核心改变：PIN 降级为身份验证工具

密钥强度不再依赖 PIN，而依赖椭圆曲线离散对数（128-bit 安全）。PIN 只用于验证"对方不是我冒充的"。

## 二、ECDH P-256 直觉

双方各生成私钥(256-bit随机数) → 各自计算公钥 → 交换公钥 → 分别计算 DHKey = ECDH(自己的私钥, 对方的公钥)。攻击者即使抓到双方公钥，也无法算出 DHKey（椭圆曲线离散对数难题）。

## 三、流程对比 Legacy

| | Legacy | SC |
|---|--------|-----|
| Phase 2 | PIN→TK→AES-CMAC→STK | ECDH 交换公钥→DHKey |
| Phase 3 | STK加密→分发LTK | 直接从DHKey生成LTK |
| 验证 | 无(PIN即验证) | Numeric Comparison确认码 / OOB |

## 四、F4/F5/F6 工具箱

| 函数 | 输入 | 输出 | 用途 |
|------|------|------|------|
| f4 | 公钥,随机数 | 128-bit | Numeric Comparison 6位确认码 |
| f5 | DHKey,随机数,地址 | LTK+MacKey | 生成 LTK |
| f6 | MacKey,随机数,IO,地址 | 128-bit | 校验配对完整性 |

DHKey→f5→LTK(加密链路用) + f4→确认码(用户比对用) → f6(完整性校验)

## 五、Numeric Comparison 防 MITM 原理

双方基于公钥各自计算确认码。MITM 攻击者必须用自己的公钥替换双方公钥→双方算出不同的确认码→用户发现数字不同→拒绝。

## 六、SC Just Works vs SC Numeric Comparison

| | SC Just Works | SC NumComp |
|---|-------------|-----------|
| 密钥强度 | 128-bit ECDH | 128-bit ECDH |
| 防被动窃听 | ✅ | ✅ |
| 防主动 MITM | ❌ 无验证 | ✅ 用户比对 |

SC Just Works 虽不防 MITM，但比 Legacy Just Works 安全——ECDH 保证即使被动窃听也无法解密。

## 七、SC 解决的 Legacy 缺陷

- Just Works 被动窃听 → SC 防住了（ECDH）
- Passkey 20-bit → SC 128-bit（不依赖 PIN）
- 唯一未解决：前向安全性（LTK 静态，泄露后历史通信可解密）

## 问答记录

### Q1：公私钥交换在哪一步？密钥是怎么生成的？数字确认到底做了什么？

**BLE SC 加密完整五阶段流程**：

```
Phase 1: 配对特征交换     ─→ SMP Pairing Req/Rsp，交换 IO Capabilities
Phase 2: 公钥交换         ─→ SMP Public Key PDU，明文交换 ECDH P-256 公钥
Phase 3: 认证             ─→ Numeric Comparison，用户比对 6 位数字
Phase 4: LTK 生成         ─→ ECDH 计算 DHKey → f5 生成 LTK
Phase 5: 链路加密启动     ─→ LL_ENC_REQ/RSP → Session Key → AES-CCM 加密
```

#### Phase 1：配对特征交换

```text
Phone(KeyboardDisplay)                 耳机(DisplayYesNo)
  ├── SMP Pairing Request ──────────────→│
  │   IO Cap = KeyboardDisplay            │
  │   AuthReq = SC | Bonding              │
  │←── SMP Pairing Response ─────────────┤
  │   IO Cap = DisplayYesNo               │
双方确认：都有显示 → 用 Numeric Comparison
```

#### Phase 2：公钥交换（公私钥在这一步交换!）

```text
Phone:                                  耳机:
  生成 256-bit 私钥 sk_a                   生成 256-bit 私钥 sk_b
  Pka = sk_a × G (椭圆曲线点乘)             Pkb = sk_b × G

  ├── SMP Public Key: Pka ──────────────→│
  │←── SMP Public Key: Pkb ─────────────┤

双方各持有: (自己的私钥, 对方的公钥)
公钥通过 SMP 走 L2CAP(CID=0x0006) 明文传输。
明文没关系——ECDH 安全性不依赖公钥保密，依赖"从公钥反推私钥数学不可行"。
```

#### Phase 3：Numeric Comparison（数字确认做了什么）

**3a：各自计算 6 位确认码**

```text
Phone:                 耳机:
C_a = f4(Pka, Pkb, Na)  C_b = f4(Pka, Pkb, Nb)
双方各显示 6 位数字 → 用户肉眼比对 → 确认一致
```

**3b：为什么防 MITM**

```text
正常：双方公钥相同    → f4 算出同一数字 → 用户看到一致 ✓
MITM：攻击者替换公钥  → f4 算出的数字不同 → 用户看到不一致 → 拒绝 ✗
```

f4 的数学性质保证：公钥不同则数字必不同。用户在此替代了 CA 的角色——亲眼见证公钥没被掉包。

#### Phase 4：LTK 生成（密钥怎么生成）

```text
DHKey = ECDH(自己的私钥, 对方的公钥)
     = sk_a × sk_b × G  ← 两边算出的是同一个值！

密钥派生：
DHKey ──f5──→ LTK (128-bit)        ← 长期密钥，存绑定表
       ──f5──→ MacKey (128-bit)     ← 用于完整性校验
       ──f6──→ 校验值                ← 验证配对过程没被篡改

完整派生链：
私钥→公钥→ECDH→DHKey→f5→LTK→AES-CMAC(+SKDm||SKDs)→SessionKey
                                          ↑每次连接全新随机数
```

#### Phase 5：链路加密启动

```text
LL_ENC_REQ/RSP → 交换 SKDm/SKDs
SessionKey = AES-CMAC(LTK, SKDm||SKDs)  ← 每条连接都不同
LL_START_ENC_REQ/RSP → 之后全部 AES-CCM 加密
```

**完整流程图**：

```text
┌────────────────────────────────────────────────────────────────────┐
│                     BLE SC 加密全流程                                │
├────────┬───────────────────────────────────────────────────────────┤
│ Phase1 │ SMP Pairing Req/Rsp → 交换 IO Capabilities                │
│        │ → 决定用 Numeric Comparison 还是 Just Works                │
├────────┼───────────────────────────────────────────────────────────┤
│ Phase2 │ 各自生成 ECDH P-256 密钥对                                  │
│        │ SMP Public Key PDU → 明文交换公钥 ← 公私钥在这里交换！       │
├────────┼───────────────────────────────────────────────────────────┤
│ Phase3 │ f4(Pka, Pkb, Na) → 各自算出 6 位数字                       │
│        │ 用户肉眼比对 → 确认身份 ← 用户替代 CA！                      │
├────────┼───────────────────────────────────────────────────────────┤
│ Phase4 │ ECDH(sk_a, Pkb) = DHKey ← 椭圆曲线魔法                    │
│        │ f5(DHKey, ...) → LTK ← 密钥在这里生成！                     │
│        │ f6(MacKey, ...) → 完整性校验                               │
├────────┼───────────────────────────────────────────────────────────┤
│ Phase5 │ LL_ENC_REQ/RSP → SessionKey = AES-CMAC(LTK, SKDm||SKDs)   │
│        │ LL_START_ENC → 加密开启，之后全用 AES-CCM                   │
└────────┴───────────────────────────────────────────────────────────┘
```

### Q2：公私钥交换涉及广播包吗？

**不涉及。** 整个配对和加密过程都在连接建立之后，走 LL Data PDU（L2CAP CID=0x0006 的 SMP 信道）。

```
广播阶段：ADV_IND → SCAN_REQ → SCAN_RSP → CONNECT_IND
          ↑ 31 字节，装不下 64 字节的 ECDH 公钥
          ↑ 没有 L2CAP 信道，无法做 SMP 交互

连接阶段：═══ Connection ═══
          SMP 开始 → 7 个 PDU 完整交互
          LL Data PDU 承载一切
```

**广播只负责"发现"和"发起连接"，之后全部安全操作都在 LL Data PDU 里。**

### Q3：SC Just Works 下，数据加密了吗？中间人能解密吗？

**加密了，但 MITM 攻击者能解密。** 加密和认证是两个独立概念：

| | SC Just Works | SC Numeric Comparison |
|---|---|---|
| **密钥强度** | 128-bit ECDH | 128-bit ECDH |
| **被动窃听** | ❌ 解不开 | ❌ 解不开 |
| **主动 MITM** | ✅ **能解密！被掉包的公钥不知道自己被掉包** | ❌ 数字对不上，用户拒绝 |

**MITM 攻击 SC Just Works 的原理**：

```text
Phone ──Pka──→ [M 用自己的 Pk_m 替换] ──Pk_m──→ 耳机
Phone ←─Pk_m'─ [M 也冒充耳机]          ←─Pkb─── 耳机

结果：
  Phone ↔ M:  DHKey₁ → SessionKey₁ → M 能解密 Phone 的所有数据
  M ↔ 耳机:   DHKey₂ → SessionKey₂ → M 能解密耳机的所有数据
  
  M 完全透明地读、改、转发一切
```

**对比 Legacy Just Works**：被动窃听者也能解密（TK=0，STK 极易算出），SC 至少需要攻击者主动介入配对。SC 把攻击门槛从"旁听就能破"提升到了"必须拦截配对过程"。

**流量计实践**：初次绑定用 Numeric Comparison 确保身份，之后每次重连用已存储 LTK 加密——攻击者没参与过初次密钥交换，拿不到 LTK，后续加密就解不开。
