# 5.6 链路层加密

## 一、加密位置

链路层加密，对上层 L2CAP→ATT→GATT 完全透明。上层只管收发，链路层自动加解密。

## 二、AES-CCM

同时提供加密和完整性校验：
- **CTR（Counter Mode）**：加密（明文→密文）
- **CBC-MAC**：生成 MIC（4字节防篡改标签）

接收方：CRC校验 → AES-CTR解密 → MIC比对。MIC 不匹配 = 数据被篡改或密钥不对 → 丢弃。

## 三、Session Key 派生

LTK 不直接用于加密，而是派生 Session Key：
```
Session Key = AES-CMAC(Key=LTK, Data=SKDm||SKDs)
```

每次连接 SKDm/SKDs 是全新随机数 → Session Key 每次都不同 → 防重放攻击。

## 四、加密三次握手

```
LL_ENC_REQ (SKDm, IVm, Rand, EDIV)
LL_ENC_RSP (SKDs, IVs)
LL_START_ENC_REQ ← 明文！此后所有包加密
LL_START_ENC_RSP
═════ 后续全部加密 ═════
```

LL_START_ENC_REQ 本身明文（对方需先读到"开始加密"才能切换模式）。

## 五、加密范围

| 字段 | 加密？ |
|------|-------|
| Header（LLID/SN/NESN/MD/Length）| ❌ 明文（链路层必须先解析） |
| Payload（L2CAP+ATT 数据）| ✅ 密文 |
| MIC | ✅ 密文 |
| CRC | ❌ 明文（物理层误码检测） |

## 六、对开发的影响

- 加密透明，App 不控制开关
- Insufficient Encryption 错误 = 链路未加密 → 检查绑定状态
- MIC 失败 = 密钥不匹配 → `removeBond()` 重新配对
- LL_ENC_REQ/RSP 在 HCI Snoop Log 中可见
- 空包在加密状态下也会加密
