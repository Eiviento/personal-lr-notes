# 5.3 LE Legacy Pairing 详解

## 一、核心问题

如何用很弱的 PIN（6位 ≈ 20 bit）生成可用于链路加密的密钥？

答案：TK + 明文随机数(Mrand/Srand) → AES-CMAC → STK（一次性密钥）→ 用 STK 加密链路 → 分发 LTK（长期密钥）。

## 二、完整流程

**Phase 1**：明文交换 Pairing Request/Response（IO Cap, OOB, AuthReq 等参数）

**Phase 2**：生成 STK
- TK 来源：Just Works=0 / Passkey=PIN / OOB=NFC传输的128-bit
- STK = AES-CMAC(Key=TK, Data=Mrand||Srand)

**Phase 3**：用 STK 加密链路 → 分发 LTK+EDIV+Rand → 绑定

## 三、TK 安全性

| 配对方式 | TK | 攻击者能算 STK？ |
|---------|-----|----------------|
| Just Works | 全0(已知) | ❌ 能，Mrand/Srand 明文 |
| Passkey | PIN(未知) | ✅ 不能，只有 1/10⁶ 猜中 |
| OOB | 随机数(NFC传输) | ✅ 不能 |

## 四、EDIV + Rand

LTK 的索引。Slave 绑定表里存多个 LTK（多台手机），下次连接时 Master 发 EDIV+Rand，Slave 查表取出对应 LTK。

## 五、安全缺陷

| 缺陷 | 原因 |
|------|------|
| Just Works 不防 MITM | TK=0 + 随机数明文 = 攻击者可算 STK |
| Passkey 仅 20 bit | 6位PIN → 10⁶ 暴力可行 |
| 无前向安全性 | LTK 静态，泄露则历史通信全解密 |
| PIN 是安全锚点 | 无法达到 128-bit 安全 |

→ 蓝牙 4.2 引入 LE Secure Connections 解决这些问题。
