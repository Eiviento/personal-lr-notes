# 5.5 密钥分发与绑定

## 一、密钥分发

配对完成后（链路已加密），双方互相分发三把钥匙：

| 钥匙 | 用途 |
|------|------|
| LTK (128-bit) | 加密链路，下次连接直接跳过配对 |
| IRK (128-bit) | 解析 Resolvable Private Address |
| CSRK (128-bit) | 数据签名（防篡改，不加密内容） |

附带：EDIV(16-bit) + Rand(64-bit) 作为 LTK 的索引，Slave 查绑定表用。

## 二、BLE 地址类型

```
Public Address — IEEE 分配，全球唯一，永久不变
Random Address
├── Static Random — 上电随机，本次不变，下次可换
└── Private Address
    ├── Non-Resolvable — 完全随机，无法追踪
    └── Resolvable — 基于 IRK+prand 生成，每 15 分钟换
                     绑定设备用 IRK 能认出
```

## 三、Resolvable Private Address 机制

地址 = [hash(24bit)][prand(24bit)]，hash = AES-128(Key=IRK, Data=prand) 低 24 位。

绑定过的手机收到地址后，用 IRK 重新算 hash，和地址中的 hash 比较 → 一致就是已绑定设备。外人看到的只是每 15 分钟换一次的随机地址，无法追踪。

## 四、绑定表结构

每条记录包含：Peer 地址、LTK、EDIV、Rand、IRK、CSRK、安全等级、密钥长度。通常存 5~10 条（Flash 容量限制）。

## 五、绑定后快速重连

连接 → Master 发 EDIV+Rand → Slave 查绑定表取 LTK → LL_ENC_REQ/RSP → LL_START_ENC → 链路加密 → 直接 GATT 交互。用户零感知，无弹窗。

## 六、Android API

`createBond()` / `removeBond()` / `getBondedDevices()` / `ACTION_BOND_STATE_CHANGED`

## 七、问答：为什么 BLE 没有 TLS 那样的 CA 证书？

**Q: ECDH 密钥交换为什么不需要 CA 公钥？怎么保证对面就是我要连的设备？**

BLE 和 TLS 的安全模型完全不同：

**TLS 模型**：依赖 PKI（CA 证书链），你信任 CA → CA 签名证明"这个服务端确实是 bank.com" → 再做 DH。信任锚是远在天边的第三方。

**BLE 模型**：依赖物理邻近 + 用户肉眼。没有证书体系，DH 密钥交换不防 MITM，但 Numeric Comparison 让双方显示相同的 6 位数字 → 你亲眼看见、亲手比对、手动确认 → 信任锚是你自己。

BLE 不用 CA 的原因：设备无互联网（不能查 CRL/OCSP）、场景是离线点对点（人和设备在一起）、IoT 规模太大（百万设备发证书不可行）、计算和功耗限制。

**一句话**：TLS 用 CA 签名解决"对面是谁"，BLE 用你眼睛解决。你本人 = BLE 的 CA。
