### 第二部分：具体视频格式

#### K06 — YUY2 深度解析 ★
> 前置：K02, K03

- YUY2 是什么：一种 packed 4:2:2 YUV 像素格式
- 内存布局：`Y₀ U₀ Y₁ V₀ | Y₂ U₂ Y₃ V₂ | ...`
- 每 4 字节 = 2 个像素，每像素平均 16 bit
- YUY2 vs 其他 YUV 排列（planar / packed / semi-planar）
- 为什么 YUY2 不是"编码格式"而是"像素格式"
- 典型来源：USB 摄像头、采集卡、DirectShow
- 为什么采集设备偏好 YUY2：低延迟、无压缩损耗、逐帧独立

#### K07 — MJpeg 深度解析 ★
> 前置：K01, K04

- JPEG 压缩原理回顾（颜色空间转换→DCT→量化→熵编码）
- MJpeg = 每一帧都是一张独立 JPEG 图片
- 帧内编码（Intra-frame）的含义
- 优势：帧独立→任意帧随机访问、编辑友好、单帧丢失不扩散
- 劣势：同画质下文件 ≈ H.264 的 3~5 倍
- 典型应用：监控摄像头、无人机图传、医学影像、编辑中间格式

#### K08 — H.264 / AVC 基础架构 ★
> 前置：K04, K05

- H.264 在编码标准谱系中的位置
- H.264 = ITU-T H.264 = ISO/IEC 14496-10（AVC）
- 核心思想：空间冗余 + 时间冗余双重压缩
- 编码器流程：预测→变换→量化→熵编码
- 解码器流程：熵解码→反量化→反变换→重建
- NAL Units：H.264 数据的基本单元
- SPS 和 PPS：解码器初始化必须的元信息

#### K09 — GOP 结构与 I/P/B 帧 ★
> 前置：K08

- GOP（Group of Pictures）= 独立解码单元
- I 帧（Intra/IDR）：完整编码独立帧
- P 帧（Predicted）：只存与前面参考帧的差异，只能向前参考
- B 帧（Bidirectional）：参考前后两帧，压缩率最高
- 典型 GOP：`I B B P B B P B B P ...`
- GOP 长度的影响：短 GOP 低延迟高码率 vs 长 GOP 高压缩高延迟
- 场景选择：监控录像短 GOP、电影分发长 GOP

#### K10 — H.264 编码工具深度 ★
> 前置：K09

- 帧内预测：同一帧内用相邻已编码块预测当前块
- 帧间预测：运动估计 + 运动补偿
- 运动向量：只记录"块移动了多少"
- 残差编码：预测值与真实值的差
- 变换与量化：DCT→量化（QP 控制质量）
- CABAC vs CAVLC：两种熵编码
- 去块效应滤波

#### K11 — H.264 Profile 与 Level ★
> 前置：K08

- Profile 定义可用编码工具集（Baseline/Main/High/Extended...）
- Level 定义解码能力上限（分辨率/帧率/码率/参考帧数）
- Profile + Level = 编解码兼容性保证
- 实际使用：网络视频 High@L4.0，4K 用 High@L5.1

#### ☆K12 — H.264 vs H.265 vs AV1 对比
> 前置：K08, K09

- H.265：同画质码率减半，编码更复杂，从宏块到 CTU
- AV1：开源免专利，压缩更好但编码极慢
- 生态现状：H.264 兼容性之王，H.265 用于 4K/HDR，AV1 在推广中

> 前置：K05

#### K14 — 为什么转封装有时会失败 ★

- 时间基（Timebase）不兼容

#### K15 — 转码（Transcoding）完整解析 ★
> 前置：K08, K13

- 完整流程：解码器→原始像素→编码器
- 为什么必然有损：量化不可逆
- 代际损失：每次转码画质都降
- 六大场景：格式转换/分辨率缩放/帧率转换/码率调整/画面处理/色彩转换
- FFmpeg 命令：`ffmpeg -i input.mp4 -c:v libx264 -crf 23 -preset medium -c:a aac output.mp4`

#### K16 — 软件编码器 vs 硬件编码器 ★
> 前置：K15

**软件编码器（x264、x265、libvpx）— 跑在 CPU 上**

工作方式：通用 CPU 指令执行编码算法。所有计算都在 CPU 核心上完成。

优点：
- 画质最好（同码率下最优）：可以做最充分的 motion estimation 搜索
- 参数灵活：preset、tune、profile、level 全部可控
- 更新方便：代码升级即可，不依赖硬件

缺点：
- 慢：1080p@30fps 实时编码需要大量 CPU。x264 medium ≈ 3-10x 实时，x265 medium ≈ 可慢到 0.3x 实时
- 耗电：CPU 满负载功耗高

**硬件编码器（NVENC / QSV / AMF / VideoToolbox）— 专用 ASIC 芯片**

工作方式：GPU 或 CPU 内集成的专用视频编码电路，固定功能硬件，不走通用计算流水线。

优点：
- 极快：通常远超实时（1080p@30fps 可跑到几百 fps）
- 低功耗：专用电路功耗远低于 CPU 全速运转
- 不占 CPU：CPU 可以做其他事情

缺点：
- 同码率画质略差于软编码（约差 5-15%）：硬件做不了太复杂的搜索
- 参数可调范围小：很多 preset/tune 选项不支持
- 芯片绑定：不同代芯片的编码质量不同（NVENC 的 Turing 架构比 Kepler 好很多）

**各平台硬编码器对照**

| 厂商 | 编码器 | FFmpeg H.264 名称 | FFmpeg H.265 名称 |
|------|--------|-------------------|-------------------|
| NVIDIA | NVENC | `h264_nvenc` | `hevc_nvenc` |
| Intel | QuickSync (QSV) | `h264_qsv` | `hevc_qsv` |
| AMD | AMF/VCE | `h264_amf` | `hevc_amf` |
| Apple | VideoToolbox | `h264_videotoolbox` | `hevc_videotoolbox` |

**选型原则**

| 场景 | 选什么 | 原因 |
|------|--------|------|
| 实时直播推流 | 硬编码 | 速度优先、CPU 留给其他任务 |
| 视频归档存储 | 软编码(x264 veryslow) | 画质/文件大小优先 |
| 视频会议 | 硬编码 | 低延迟、多路并发的场景 |
| 视频压制发布 | 软编码 | 最终画质最重要 |
| 监控多路录像 | 硬编码 | 多路并发（16 路 1080p），CPU 不够用 |
| 云转码服务 | 软编码集群 | 多机器并行，画质和灵活性优先 |

**硬编码质量控制**

硬件编码器的质量在不断提升：
- NVIDIA NVENC：Turing 架构(2018)开始画质接近 x264 medium，Ada(2022)进一步追近
- Intel QSV：第 11 代(Tiger Lake)以后画质大幅改善
- Apple M 系列芯片的 VideoToolbox 质量优秀

但软编码的同码率画质仍然保持领先，尤其是 x265 veryslow 这种极端档位。

#### ☆K17 — 码率控制深入（CRF/CBR/VBR/ABR）
> 前置：K04, K15

**CRF 深入**

x264 的 CRF 取值区间是 0~51（x265 是 0~51），实际使用时：
- CRF 0 = 无损模式（文件巨大，仅特殊用途）
- CRF 16-18 = 视觉近乎无损，适合母带存档（码率可能达到 50+ Mbps @1080p）
- CRF 19-22 = 高质量，适合画质敏感的发布（码率约 15-30 Mbps @1080p）
- **CRF 23 = 默认值，质量和大小最佳平衡**（码率约 8-15 Mbps @1080p）
- CRF 24-26 = 中等质量，适合一般用途
- CRF 27-30 = 低质量但文件小，适合移动端或预览
- CRF 35+ = 明显画质下降，仅适合极端省空间

同一个 CRF 值码率因内容而异：动画片 CRF 23 可能 3-5 Mbps，动作大片 15-20 Mbps。

**2-Pass VBR**

原理：
- 第一遍（Pass 1）：编码器跑一遍全片但不输出最终文件，只统计每一帧/每个 GOP 的复杂度分布
- 第二遍（Pass 2）：根据第一遍的统计，精确分配码率——简单场景给少、复杂场景给多，确保最终平均码率达标

场景：需要精确控制输出文件大小时（如蓝光光盘必须 ≤ 光盘容量，流媒体平台要求固定平均码率）

```bash
# 2-Pass VBR
ffmpeg -i input.mp4 -c:v libx264 -b:v 5M -pass 1 -f mp4 /dev/null
ffmpeg -i input.mp4 -c:v libx264 -b:v 5M -pass 2 output.mp4
```

**CBR + VBV（直播场景）**

VBV（Video Buffering Verifier）模拟一个解码缓冲区，确保瞬时码率峰值不超过缓冲上限：

```bash
# 直播用 CBR + VBV
ffmpeg -i input -c:v libx264 -b:v 5M -maxrate 5M -bufsize 10M output.flv
# -maxrate: 瞬时码率上限
# -bufsize: 解码缓冲区大小（越大允许的瞬时波动越大）
```

**选择决策**
- 不知道输出文件多大 → CRF（最省心）
- 必须控制文件大小在某个精确值 → 2-Pass VBR
- 推流需要稳定码率 → CBR + VBV

#### K18 — 关键编码参数实战（preset/tune/profile） ★
> 前置：K11, K15

**x264 preset — 速度与压缩效率的 10 档**

| preset | 速度（相对） | 压缩效率 | 适用场景 |
|--------|-----------|---------|---------|
| ultrafast | 最快 | 最低 | 实时编码、屏幕录制 |
| superfast | | | |
| veryfast | | | 直播编码、视频会议 |
| faster | | | |
| fast | | | |
| **medium** | **基准** | **基准** | **默认档，通用** |
| slow | | | 压制发布 |
| slower | | | |
| veryslow | 最慢 | 最高 | 归档存储、母带 |

ultrafast vs veryslow 的差异：
- 编码速度：可能差 10-50 倍
- 同样画质下文件大小：可能差 30-50%（veryslow 文件小得多）
- 差异来自：运动估计搜索范围、参考帧数、分区方式、RDO 优化等

**x264 tune — 针对内容类型的优化**

| tune | 针对 | 调了什么 |
|------|------|---------|
| film | 实拍视频 | 去块滤波稍强、psy-rd 优化 |
| animation | 动画片 | 更多参考帧（动画帧重复多）、去块滤波不同 |
| grain | 胶片颗粒感 | 保留颗粒而不是抹掉 |
| stillimage | 幻灯片/静态图 | 长 GOP、多参考帧 |
| zerolatency | 低延迟 | 无 B 帧、无 lookahead、slice-based 编码 |
| fastdecode | 解码速度 | 禁用 CABAC、禁用去块滤波 |
| psnr / ssim | 客观指标优化 | 禁用 psy-rd——不建议用 |

**组合实战**

```bash
# 高质量归档
ffmpeg -i input -c:v libx264 -crf 18 -preset veryslow -tune film output.mp4

# 快速压制发布
ffmpeg -i input -c:v libx264 -crf 23 -preset fast output.mp4

# 低延迟推流
ffmpeg -i input -c:v libx264 -preset ultrafast -tune zerolatency -b:v 5M output.flv

# 动画压制
ffmpeg -i input -c:v libx264 -crf 23 -preset slow -tune animation output.mp4
```

### 第四部分：格式间的转换实战

#### K19 — YUY2 → H.264 转换全链路 ★
> 前置：K06, K08

这是一条把前面所有知识串起来的实战链路——从 USB 摄像头拿到 YUY2 原始帧，到最终输出可以播放的 MP4 文件。

**完整数据流**

```
USB 摄像头 → YUY2 原始帧 → 像素格式转换 → H.264 编码 → MP4 封装 → 文件/推流
    ↑             ↑              ↑             ↑            ↑
  K06 学过的   每帧 ~4.15MB    4:2:2→4:2:0   K08-K11     K05+K13
  采集来源     1080p          色度采样转换    编码流程     封装
```

**详细步骤**

步骤 1 — 获取 YUY2 帧：从采集设备读原始帧数据（DirectShow / V4L2 / AVFoundation）
步骤 2 — 色度采样转换（4:2:2 → 4:2:0）：H.264 主流用 4:2:0，YUY2 是 4:2:2，需下采样 UV。常用 bilinear 或 lanczos 算法，质量有差异
步骤 3 — H.264 编码：设置编码器（x264 或 NVENC）、CRF/码率、preset、GOP 长度、Profile/Level
步骤 4 — MP4 封装：编码后的 H.264 裸流 + AAC 音频 → 封装为 MP4

**数据量对比**

| 阶段 | 格式 | 1080p@30fps 数据速率 |
|------|------|---------------------|
| 采集 | YUY2 原始 | ~1 Gbps |
| 采样转换后 | YUV 4:2:0 原始 | ~746 Mbps |
| 编码后 | H.264 CRF 23 | ~5-15 Mbps |

压缩了约 66-200 倍。

**FFmpeg 命令**

```bash
# Windows DirectShow 采集 YUY2 + 编码
ffmpeg -f dshow -pixel_format yuyv422 \
       -i video="USB Camera" \
       -c:v libx264 -crf 23 -preset veryfast \
       -c:a aac -b:a 128k \
       output.mp4
```

**关键点**
- 4:2:2 → 4:2:0 的下采样策略影响画质（尤其是在红色文字/细线场景）
- 如果采集源本身就是 4:2:0 的摄像头，用 NV12 而非 YUY2 更高效
- 采集链路的瓶颈通常是 USB 带宽和编码器速度

#### K20 — MJpeg → H.264 转码实战 ★
> 前置：K07, K08

目标：把一个 MJpeg 格式的视频转成 H.264，大幅减少文件体积。

**为什么能大幅压缩**

MJpeg = 全 I 帧，每帧独立 JPEG。H.264 = I+P+B 帧，大多数帧只存储变化量。

同样内容（比如 30 秒的监控视频，画面基本静止）：
- MJpeg：30fps × 30s = 900 帧完整 JPEG → ~500 MB
- H.264（典型 GOP=30）：30 个 I 帧 + 870 个 P/B 帧 → ~50 MB
- 压缩约 10 倍（静止画面更多）

**收益评估**

| 内容类型 | MJpeg 大小 | H.264(CRF 23) | 压缩比 |
|---------|-----------|---------------|--------|
| 静止监控画面 | 500 MB | 20 MB | ~25× |
| 正常活动画面 | 500 MB | 100 MB | ~5× |
| 高速运动 | 500 MB | 200 MB | ~2.5× |

**代价**

- 失去逐帧编辑能力：MJpeg 可以在任意帧切割，H.264 只能在 I 帧处精准切割
- seek 变慢：H.264 需要回溯最近 I 帧并解码中间帧
- 转码耗时：软编码 10 分钟视频需要 3-5 分钟（x264 medium）

**如果仍需帧独立性**

```bash
# -g 1 = GOP=1 = 全 I 帧 H.264
ffmpeg -i input.mjpeg -c:v libx264 -g 1 -crf 23 output.mp4
# 仍然比 MJpeg 小（H.264 的 I 帧压缩比 JPEG 更高效），但比正常 H.264 大得多
```

**FFmpeg 命令**

```bash
# 标准转换
ffmpeg -i input.avi -c:v libx264 -crf 23 -preset medium \
       -c:a aac -b:a 128k output.mp4

# 如果 MJpeg 在 AVI 容器中，可能需要指定输入
ffmpeg -f mjpeg -i input.mjpeg -c:v libx264 -crf 23 output.mp4
```

#### K21 — 不同格式应用场景全景对比 ★
> 前置：K06, K07, K08, K13, K15

**YUY2 / MJpeg / H.264 全维度对比**

| 维度 | YUY2 | MJpeg | H.264 |
|------|------|-------|-------|
| 类型 | 原始像素格式 | 帧内编码 | 帧内+帧间编码 |
| 层级 | 数据排列方式 | 压缩编码 | 压缩编码 |
| 色度采样 | 4:2:2 | 4:2:2 或 4:2:0 | 通常 4:2:0 |
| 每像素 bit | 16 | 变（压缩后） | 变（压缩后） |
| 压缩率 | 无压缩 | 1:10 ~ 1:20 | 1:50 ~ 1:200 |
| 1080p@30fps 码率 | ~1 Gbps | 50-100 Mbps | 5-10 Mbps |
| 帧独立性 | ✅ 每帧独立 | ✅ 每帧独立 | ❌ P/B 帧依赖 I 帧 |
| 随机访问 | 任意帧 | 任意帧 | 只能从 I 帧 |
| 延迟 | 零延迟 | 低延迟（毫秒级） | 取决于 GOP 长度 |
| 编辑友好度 | ★★★ | ★★★ | ★☆（需在 I 帧处切） |
| 容错性 | — | 强（丢帧不扩散） | 弱（P 帧丢→错误扩散） |
| 兼容性 | DirectShow 生态 | 无统一标准 | ★★★ 所有设备 |
| 硬件成本 | 极低（无编码芯片） | 低（JPEG 芯片） | 需编码芯片或 CPU |

**场景决策树**

```
你是做什么的？

采集端（摄像头/采集卡）
├─ 视频会议 → YUY2/NV12 采集 → H.264 硬编码 → 推流
├─ 直播 → YUY2 采集 → H.264 硬编码(zerolatency) → RTMP
└─ 录制存档 → YUY2 采集 → H.264 软编码(CRF 18) → MP4 文件

存储/分发
├─ 长期归档 → H.264/H.265 CRF 18-23
├─ 在线视频 → H.264 High@L4.0
├─ 4K 内容 → H.265 High@L5.1
└─ 兼容老设备 → H.264 Baseline@L3.0

编辑制作
├─ 专业后期 → ProRes / DNxHD（帧内编码专业格式）
├─ 简单剪辑 → MJpeg / 全 I 帧 H.264（-g 1）
└─ 预览版 → H.264 CRF 23（通用）

实时预览
├─ 监控实时画面 → MJpeg（低延迟 + 丢帧不扩散）
├─ 无人机图传 → MJpeg（信号中断秒恢复）
└─ 医学影像 → MJpeg / YUY2 原始（每帧必须绝对准确）

带宽受限
├─ 极低带宽 → H.265/AV1 + 低分辨率
└─ 普通网络 → H.264 + CRF 23-28
```

#### K22 — FFmpeg 入门：基础命令与思维模型 ★
> 前置：K05

**FFmpeg 命令通用结构**

```
ffmpeg [全局选项] -i [输入文件] [输入选项] [输出选项] [输出文件]

全局选项：-y(覆盖输出) -hide_banner(隐藏版本信息) -loglevel
输入选项：-ss(起始时间) -t(持续时间) -r(覆盖帧率)
输出选项：-c:v(视频编码器) -c:a(音频编码器) -vf(视频滤镜) -b:v(视频码率)
```

**filter graph 思维模型**

```
简单滤镜（-vf / -af）：单输入 → 单输出，线性链
    -vf "scale=1280:720,format=yuv420p"
        输入 → scale → format → 输出

复杂滤镜（-filter_complex）：多输入多输出，有向图
    -filter_complex "[0:v][1:v]overlay=10:10[out]"
        视频流0 ─┐
                   ├→ overlay → [out]
        图片流1 ─┘
```

**流映射（-map）**

一个文件可能有多个流（视频/音频/字幕），-map 控制选哪些：

```bash
# 选择第 0 个输入的第 0 条视频流 + 第 0 条音频流
ffmpeg -i input.mp4 -map 0:v:0 -map 0:a:0 -c copy output.mp4

# 选择所有流
ffmpeg -i input.mp4 -map 0 -c copy output.mp4
```

**命令速查表**

| 操作 | 命令 |
|------|------|
| 查看文件信息 | `ffprobe -v quiet -print_format json -show_format -show_streams file.mp4` |
| 转封装（MKV→MP4） | `ffmpeg -i in.mkv -c copy out.mp4` |
| 转码（H.264 CRF） | `ffmpeg -i in.mp4 -c:v libx264 -crf 23 -preset medium out.mp4` |
| 转码+缩放 | `ffmpeg -i in.mp4 -vf scale=1280:720 -c:v libx264 -crf 23 out.mp4` |
| 降低帧率 | `ffmpeg -i in.mp4 -r 30 -c:v libx264 -crf 23 out.mp4` |
| 截取片段 | `ffmpeg -ss 00:01:00 -t 00:00:30 -i in.mp4 -c copy out.mp4` |
| 硬件编码(NVIDIA) | `ffmpeg -i in.mp4 -c:v h264_nvenc -cq 23 out.mp4` |
| moov 移到文件头 | `ffmpeg -i in.mp4 -c copy -movflags +faststart out.mp4` |
| 提取音频 | `ffmpeg -i in.mp4 -vn -c:a copy out.aac` |
| 去除音频 | `ffmpeg -i in.mp4 -c:v copy -an out.mp4` |
| 多文件合并 | 先写 list.txt，再 `ffmpeg -f concat -i list.txt -c copy out.mp4` |

**思维模型**

FFmpeg 做的是"图处理"：输入 → 解码器 → 滤镜图 → 编码器 → 输出。每个环节用不同的参数控制。理解了 `输入→滤镜→编码→输出` 这条线，就能组合出任意命令。

#### K23 — 用 MediaInfo 和 FFprobe 分析视频文件 ★
> 前置：K05

拿到一个未知视频文件，用这两把"手术刀"看透它。

**MediaInfo（GUI 工具）**

打开文件后关注这些字段：
- Format/Container：容器格式（MPEG-4 = MP4、Matroska = MKV）
- Video Format/Codec ID：编码格式（AVC = H.264、HEVC = H.265）
- Width × Height：分辨率
- Frame rate：帧率
- Bit rate / Overall bit rate：码率（注意区分视频码率和总码率）
- Color space / Chroma subsampling：YUV / 4:2:0 等信息
- Format profile：Profile@Level

**ffprobe 常用命令**

```bash
# 1. 查看基本流信息（最快）
ffprobe -v error -show_entries stream=codec_name,codec_type,width,height,r_frame_rate,bit_rate input.mp4

# 2. 查看封装信息
ffprobe -v error -show_entries format=format_name,duration,bit_rate input.mp4

# 3. 以 JSON 输出完整信息
ffprobe -v quiet -print_format json -show_format -show_streams input.mp4

# 4. 查看 GOP 结构（每帧的类型）
ffprobe -v error -show_entries frame=pict_type -select_streams v:0 input.mp4 | head -50
# I=IDR帧, P=P帧, B=B帧

# 5. 只统计关键帧（I 帧）间隔
ffprobe -v error -show_entries frame=pict_type -select_streams v:0 input.mp4 | grep -c "pict_type=I"

# 6. 查看 H.264 Profile 和 Level
ffprobe -v error -show_entries stream=profile,level -select_streams v:0 input.mp4
```

**分析检查清单**

拿到一个视频文件，按此顺序检查：
1. 封装格式是什么？（format_name）
2. 视频编码是什么？（codec_name）
3. 分辨率？帧率？（width, height, r_frame_rate）
4. 码率？码率模式？（bit_rate，CBR/VBR 通过码率波动判断）
5. 像素格式/采样？（pix_fmt：yuv420p = 4:2:0，yuv422p = 4:2:2）
6. Profile 和 Level？（profile, level）
7. GOP 结构如何？（I 帧间隔 = 总帧数 ÷ I 帧数量）

**验证转封装是否无损**

```bash
# 方法 1：对比流参数 —— 应该完全一致
ffprobe -v error -show_entries stream=codec_name,width,height,duration,nb_frames input.mkv
ffprobe -v error -show_entries stream=codec_name,width,height,duration,nb_frames output.mp4

# 方法 2：提取视频裸流对比 MD5（终极验证）
ffmpeg -i input.mkv -c:v copy -f md5 -
ffmpeg -i output.mp4 -c:v copy -f md5 -
# 两个 MD5 应该完全一致 → 比特级无损
```

#### ☆K24 — 采集→编码→封装→分发：完整流水线
> 前置：K19, K20, K22

**流水线 1：USB 摄像头 → RTMP 直播推流**

```
摄像头(YUY2) → H.264 硬编码 → FLV 封装 → RTMP 推流 → 直播服务器
```

```bash
ffmpeg -f dshow -pixel_format yuyv422 -i video="USB Camera" \
       -f dshow -i audio="Microphone" \
       -c:v h264_nvenc -cq 23 -preset p1 -tune ll \
       -b:v 5M -maxrate 5M -bufsize 10M \
       -c:a aac -b:a 128k \
       -f flv rtmp://live.server.com/app/streamkey
```

参数说明：
- `h264_nvenc`：NVIDIA 硬编码，速度快
- `-preset p1 -tune ll`：NVENC 的低延迟预设
- `-b:v 5M -maxrate 5M -bufsize 10M`：CBR+VBV，直播必需的码率控制
- `-f flv`：输出格式 FLV（RTMP 只能用 FLV 容器）
- `rtmp://...`：推流地址

**流水线 2：视频文件 → HLS 切片分发**

```bash
ffmpeg -i input.mp4 \
       -c:v libx264 -crf 23 -preset medium \
       -c:a aac -b:a 128k \
       -vf "scale=-2:720" \
       -f hls -hls_time 6 -hls_list_size 0 \
       -hls_segment_filename "segment_%03d.ts" \
       playlist.m3u8
```

参数说明：
- `scale=-2:720`：缩放到高度 720，宽度按比例计算（`-2` 表示自动且保证偶数）
- `-f hls -hls_time 6`：切成 6 秒一段的 TS 分片
- `playlist.m3u8`：HLS 播放列表，播放器先读这个再按序拉 TS 分片

**流水线 3：完整转码流水线（分辨率+编码+字幕烧录）**

```bash
ffmpeg -i input_4k.mkv \
       -vf "scale=1920:1080,subtitles=chinese.srt" \
       -c:v libx264 -crf 23 -preset slow -tune film \
       -profile:v high -level:v 4.0 \
       -c:a aac -b:a 192k \
       -movflags +faststart \
       output_1080p.mp4
```

这一条命令包含了：Demux(MKV) → 解码 → 滤镜(缩放+字幕烧录) → 编码(H.264) → faststart → Mux(MP4)

#### K25 — 当你拿到一个视频文件，你应该问什么 ★
> 前置：所有前序知识点

按这 6 个问题过一遍，你就能对任何视频文件建立完整的理解：

**Q1：封装格式是什么？**
用 ffprobe 看 format_name，或者直接看文件扩展名（但不要完全信任扩展名——`.mp4` 也可能是别的格式）
→ MP4 / MKV / MOV / TS / AVI / WebM / FLV

**Q2：视频编码格式是什么？**
用 ffprobe 看 codec_name
→ H.264(avc1) / H.265(hevc) / VP9 / AV1 / MJpeg / MPEG-2
→ 决定了哪些设备能播放、是否需要转码

**Q3：像素格式/色度采样是什么？**
用 ffprobe 看 pix_fmt
→ yuv420p(4:2:0) / yuv422p(4:2:2) / yuv444p(4:4:4)
→ 决定了画质的色度精度（消费级还是专业级）

**Q4：分辨率、帧率、码率各是多少？**
用 ffprobe 看 width, height, r_frame_rate, bit_rate
→ 1080p@30fps 5Mbps = 典型在线质量
→ 4K@60fps 50Mbps = 高质量 4K

**Q5：GOP 结构如何？（对于 H.264/H.265）**
用 ffprobe 统计 pict_type
→ 全 I 帧（GOP=1）= 编辑格式
→ I 间隔 30 = 常见在线视频
→ I 间隔 250 = 高压缩分发

**Q6：Profile 和 Level 是什么？（对于 H.264/H.265）**
用 ffprobe 看 profile, level
→ High@L4.0 = 1080p 标准
→ 判断目标设备能否播放

**综合练习建议**

找 5 个不同类型的视频文件（手机录像、网上下载的电影、监控录像、直播录制、采集卡输出），用以上 6 个问题逐一分析，然后做对比。关键是养成条件反射——看到一个视频文件先问这 6 个问题。
